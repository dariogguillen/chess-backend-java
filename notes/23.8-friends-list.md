# Feature 23.8 — Friends list

**Feature ID:** `friends-list` (from `feature_list.json`)

**Status:** done

---

## What we built

Authenticated users can build a mutual friends list. Each account gets a
stable, shareable 8-char **friend code** at creation; you add someone by
entering their code (no directory, no search — the user base cannot be
enumerated). The relationship is symmetric request/accept: A sends a
request to B by code, B accepts or rejects, and either party can cancel
a sent request or remove an accepted friend. Eight new `/api/me`
endpoints cover the full lifecycle plus three paginated read surfaces
(friends, incoming requests, outgoing requests). Guest play is
untouched — friends require an account.

## Java / Spring concepts that appear

- **Hibernate ad-hoc entity joins in JPQL** — the friends/requests list
  queries resolve the *other* party's live display name with
  `JOIN User u ON u.id = f.addresseeId` (and a `CASE` expression for the
  symmetric friends list). This is an "ad-hoc" / "entity" join: `User`
  is not a mapped `@ManyToOne` association on `Friendship`, yet JPQL can
  still join two unrelated entities on an arbitrary `ON` predicate
  (supported since Hibernate 5.1). It is the mechanism that lets us
  store raw UUIDs (no relationship mapping) and *still* read live names
  in one round-trip. See `FriendshipRepository`.
- **JPQL constructor projection (`SELECT new ...`)** — `FriendSummary`
  and `FriendRequestSummary` are plain records whose canonical
  constructor JPQL invokes at result-set extraction. This keeps the
  controller free of any further entity navigation, which matters
  because the app runs with `spring.jpa.open-in-view: false` — outside
  the service transaction the entity manager is closed. Same pattern as
  feature 19's `ArchivedGamePlayerView`.
- **A database UNIQUE index as a domain invariant** — the unordered-pair
  index `UNIQUE (LEAST(requester_id, addressee_id), GREATEST(...))`
  enforces "at most one relationship per pair, in either direction" at
  the storage layer. The service pre-checks for the common case but also
  catches `DataIntegrityViolationException` on insert as the
  race-condition safety net — the same pattern `AuthService.register`
  uses for the email UNIQUE constraint.
- **`@Validated` + `@Min`/`@Max` on `@RequestParam`** — pagination
  bounds are enforced by JSR-380 constraints on the controller method
  parameters, which only fire because the class carries `@Validated`
  (plain `@Valid` covers `@RequestBody` only). A violation surfaces as
  `HandlerMethodValidationException` → `400 VALIDATION_FAILED` via the
  global handler. Mirrors `MyGamesController`.
- **Mechanical error-code derivation** — the six new exceptions extend
  the existing `NotFoundException` / `ConflictException` /
  `UnprocessableException` umbrellas, so `GlobalExceptionHandler.codeOf`
  derives their `UPPER_SNAKE_CASE` codes from the class name with **no
  new handler branch**. Only `ErrorResponse.allowableValues` and the
  `OpenApiIT` drift canary needed updating.
- **`@Transactional(readOnly = true)`** — the three list methods are
  marked read-only so Spring/Hibernate can skip dirty-checking and flush
  on those paths; the four mutating methods get a plain `@Transactional`.

## Decisions taken

- **Decision:** Friend codes are generated in one place
  (`FriendCodeGenerator`) and the column is `NOT NULL UNIQUE`, with the
  V3 migration backfilling pre-existing rows before the `NOT NULL` flip.
  - **Alternatives considered:** (a) a DB `DEFAULT` expression; (b)
    generating lazily on first read; (c) deriving the code from the user
    id.
  - **Why this one:** a single generator (mirroring `RoomCodeGenerator`)
    keeps the unambiguous alphabet and collision policy in exactly one
    location and runs at both creation paths (`AuthService.register` +
    the OAuth find-or-create). Postgres has no built-in generator for our
    exact alphabet, so a column `DEFAULT` could not match it; lazy
    generation would leave the `NOT NULL` column briefly unpopulated; a
    derived code would leak the id space. Backfilling in the migration is
    the standard "add nullable → populate → constrain" sequence for a
    `NOT NULL` column on a non-empty table.

- **Decision:** Store raw UUIDs on `friendships` (no `@ManyToOne`) and
  read live display names via entity joins — the opposite of how `games`
  snapshots the name.
  - **Alternatives considered:** (a) `@ManyToOne User` associations; (b)
    snapshot the friend's display name onto the friendship row like
    `games` does.
  - **Why this one:** `@ManyToOne` would force a lazy-association
    navigation incompatible with `open-in-view: false`, and contradicts
    the codebase's denormalised UUID house style. Snapshotting is correct
    for `games` because a game is an *audit record* (the name at game
    time is the historically accurate value), but a friendship is a
    *live relationship* — when a friend renames themselves, their friends
    should see the new name. The entity join gives us live names without
    a mapped association.

- **Decision:** Reject / cancel / remove all DELETE the row; there is no
  `REJECTED` state. `FriendshipStatus` is just `PENDING | ACCEPTED`.
  - **Alternatives considered:** keep a `REJECTED` / `CANCELLED` tombstone
    for history or to block re-requests.
  - **Why this one:** a tombstone would either block legitimate
    re-requests after a reject or require extra logic to ignore it. The
    user explicitly wanted re-requesting to be allowed, and the friends
    feature has no audit/history requirement, so the simplest closed
    two-state model wins.

- **Decision:** Acting on a request you don't participate in returns
  `404 FRIEND_REQUEST_NOT_FOUND`, never `403`.
  - **Alternatives considered:** `403 Forbidden` for "exists but not
    yours".
  - **Why this one:** a 403 confirms the request id is real, letting an
    attacker probe for valid ids. Returning the same 404 as a
    truly-missing request leaks nothing. The repository bakes the caller
    into the `WHERE` clause so the two cases are genuinely
    indistinguishable in code.

## How this compares to what I know

- **In Scala / Typelevel this would be...** the repository queries are
  the closest analogue to **doobie**, where you write the SQL join by
  hand: `sql"SELECT u.display_name FROM friendships f JOIN users u ON
  u.id = f.addressee_id WHERE ...".query[FriendSummary]`. JPQL's
  `JOIN User u ON ...` is the same explicit join, but expressed over
  *entities* rather than tables and with Hibernate generating the SQL —
  you get the doobie-style "I decide the join" control without dropping
  to raw SQL, and the `SELECT new FriendSummary(...)` projection is
  exactly doobie's `.query[FriendSummary]` row mapping. The
  `FriendshipStatus` enum is a sealed **ADT** — in Scala it would be a
  `sealed trait FriendshipStatus` with `Pending` / `Accepted` case
  objects, and the "no REJECTED state, deletes instead" decision is the
  same as choosing not to add a third case to the ADT. The unordered-pair
  UNIQUE index is a **DB-level invariant** rather than an app-level
  check: in a pure-FP service you might reach for a `Semaphore` or an
  optimistic-retry to serialize the both-directions race; here Postgres'
  expression index makes the invariant unforgeable, and the
  `DataIntegrityViolationException` catch is the functional equivalent of
  `attempt`-ing the insert and recovering the constraint failure into a
  domain error.
- **In Node this would be...** roughly an Express router with a
  `friendships` table accessed through Knex/Prisma; the entity join would
  be a Prisma `include` or a hand-written Knex `.join()`. The
  no-existence-leak rule is the same "return 404 not 403 for
  not-your-resource" idiom you'd apply in any REST API. The big
  difference is that Spring's declarative `@Transactional` +
  constraint-violation translation replaces the manual `BEGIN/COMMIT` +
  driver-error-code inspection you'd write by hand in a Node data layer.

## Gotchas / things I learned the hard way

- The `User` entity gained a `NOT NULL friend_code` column under
  `ddl-auto: validate`, which meant three test helpers that constructed
  `new User(...)` directly (bypassing the service) had to be updated to
  the 7-arg constructor with a throwaway code — otherwise the insert
  would violate `NOT NULL`. A reminder that `validate` mode makes the
  entity↔migration column contract exact.
- Spring logs a `PageImpl serialized as-is is not supported` warning when
  a controller returns a raw `Page<T>`. It's pre-existing (feature 19's
  `MyGamesController` does the same) and harmless — the wire shape is
  stable enough for this project — so I matched the existing pattern
  rather than introducing `PagedModel`.
