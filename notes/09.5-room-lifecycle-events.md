## Feature 09.5 ‚ÄĒ Room state read + real-time lifecycle events

**Feature ID:** `room-lifecycle-events` (from `feature_list.json`)

**Status:** in progress

---

## What we built

`GET /api/rooms/{id}` (REST) plus `/topic/rooms/{roomId}` (STOMP)
together unblock the frontend's `creator-game-discovery` flow.
Until this feature shipped, Player A (the creator) had no way to
learn when Player B joined the room: the `POST /api/rooms`
response carries `gameId: null` (no game yet), and the
`POST /api/rooms/{id}/join` response only reaches Player B. After
this feature, Player A subscribes to `/topic/rooms/{roomId}`
immediately after `createRoom`, receives a `RoomJoinedEvent` the
moment B joins, and transitions to `/topic/games/{gameId}`. The
GET endpoint is the polling-friendly companion and the fallback
for late subscribers (STOMP does not replay).

## Java / Spring concepts that appear

- **`sealed interface` + `permits` clause (Java 17).** `RoomEvent`
  is the supertype; the `permits RoomJoinedEvent` clause closes the
  hierarchy at the source. Like a Scala `sealed trait`, the
  compiler can prove exhaustiveness in `switch` expressions, and
  adding a new variant later requires touching the seal ‚ÄĒ there is
  no way to silently add a fourth event from elsewhere on the
  classpath. See
  [JEP 409](https://openjdk.org/jeps/409).
- **Discriminated unions in JSON via an explicit `type` field, not
  `@JsonTypeInfo`.** Jackson can write/read polymorphic types via
  `@JsonTypeInfo(use = NAME, property = "type")` and a
  `@JsonSubTypes` registry ‚ÄĒ but the discriminator becomes invisible
  in the source code. Modelling it as a normal record component
  set by a convenience constructor (`new RoomJoinedEvent(roomId,
  gameId, blackPlayer)` calls `this("ROOM_JOINED", roomId, gameId,
  blackPlayer)`) keeps the contract visible without any annotations.
  The producer always knows the variant, so the consumer-side
  reflection Jackson would otherwise do is dead weight here.
- **`SimpMessagingTemplate.convertAndSend`.** Server-side publish on
  a STOMP destination. Same template `GameService.broadcastMoveEvent`
  uses ‚ÄĒ `RoomService` constructor-injects it now too. Thread-safe;
  fire-and-forget; the Spring `MessageConverter` chain handles
  serialization. The new broadcast lives **outside** the
  `roomStore.compute` lambda (the persistence is already durable;
  a broker-side throw should not look like a failed mutation).
- **The mapper boundary pattern in `web/`.** `RoomDetailsMapper`
  is a `@Component` that turns a domain `Room` into the wire
  `RoomDetailsResponse`. It is the same shape as the
  `GameEntityMapper` we use between domain and JPA ‚ÄĒ explicit
  conversion at the layer boundary, unit-testable in isolation
  without Spring.
- **Role derivation at the boundary.** The domain `Player` has no
  `role` field; role is `players[0] = WHITE`, `players[1] = BLACK`
  inferred in `RoomDetailsMapper`. Trade-off documented in the
  decisions section.
- **STOMP fire-and-forget semantics.** A subscriber that connects
  *after* a `convertAndSend` misses the event forever ‚ÄĒ there is no
  replay. The REST GET endpoint is the catch-up path; both
  mechanisms ship together for exactly this reason.

## Decisions taken

**Ship two mechanisms (GET + STOMP) in the same feature, not one then the other.**

- Decision: introduce both the REST GET endpoint and the STOMP topic
  in the same feature.
- Alternatives: STOMP-only (sub-second push, but every late
  subscriber misses the event and would need a complementary REST
  fetch anyway); GET-only (forces 2-3s polling steady load, no
  real-time UX even when both clients are online).
- Why: the frontend repo's own `progress/current.md` lists both as
  cross-repo items #2 and #3 of its blocked work. STOMP-only
  requires perfect subscription timing ‚ÄĒ a network blip during the
  `connect+subscribe` round-trip and Player A is stuck. GET-only
  forces wasteful polling. Together they are the standard pattern
  for real-time multiplayer lobbies (Lichess, chess.com): push for
  the common path, polling fallback for the rare one.

**Discriminator on this polymorphic topic; NOT retrofitted to `MoveEvent` / `ViewerCountEvent`.**

- Decision: `/topic/rooms/{roomId}` carries a `type` field on every
  event; `/topic/games/{gameId}` (`MoveEvent`) and
  `/topic/games/{gameId}/viewers` (`ViewerCountEvent`) stay flat.
- Alternatives: retrofit `MoveEvent` and `ViewerCountEvent` with a
  `type` field "for consistency"; skip the discriminator here too and
  let consumers tell variants apart by structure.
- Why: a topic that can only ever carry one shape gains nothing from
  a discriminator ‚ÄĒ it is dead weight. A topic that **can** carry
  multiple shapes (today one, tomorrow `RoomClosedEvent`,
  `PlayerLeftEvent`) needs a stable, branch-by-string field so
  consumers don't have to do structural detection. We are stating
  this as the project's **design rule**: polymorphic topics get the
  discriminator; single-event topics don't. Documented in
  `docs/architecture.md`.

**Explicit `type` field on the record vs Jackson `@JsonTypeInfo`.**

- Decision: the `type` is a regular record component set by a
  canonical constructor; no `@JsonTypeInfo`, no `@JsonSubTypes`.
- Alternatives: Jackson's polymorphic deserialization machinery
  (`@JsonTypeInfo(use = NAME, property = "type")` on `RoomEvent`,
  `@JsonSubTypes({...})` listing each variant).
- Why: the producer always knows the variant ‚ÄĒ the convenience
  constructor sets the right constant on every emission. Jackson's
  reflection-driven approach buys nothing on the write side and
  would only matter if the consumer round-tripped through Jackson
  with the abstract `RoomEvent` static type, which it does not (the
  STOMP frontend reads it as plain JSON; the integration test
  deserializes to the concrete `RoomJoinedEvent`). Visible-in-source
  is preferable to invisible-in-annotations.

**Status enum exposed natively (`WAITING_FOR_PLAYER`, `ACTIVE`, `CLOSED`).**

- Decision: the GET response carries the `RoomStatus` enum literals
  as-is.
- Alternatives: map to a presentation vocabulary like
  `OPEN / FULL / CLOSED` or `LOBBY / PLAYING / DONE`.
- Why: the literals are already the right vocabulary ‚ÄĒ they describe
  the lifecycle in domain terms. A mapping layer would obscure
  rather than aid, and the frontend already understands the three
  states from the cross-repo design discussions. Honest to the
  domain.

**Role derived at the boundary, not added to the domain `Player`.**

- Decision: `RoomDetailsMapper` infers role from list index.
- Alternatives: add `role: Side` to the `Player` record (or a new
  `RoomMember` record with both).
- Why: keeps the domain minimal. Role is a property of the
  Player-in-a-Room, not of the Player; storing it on `Player` would
  leak the property into Redis snapshots and the Postgres history
  shape for no current benefit. The cost is that the mapper is
  position-sensitive ‚ÄĒ a future feature that rearranges
  `Room.players` would break role derivation. The
  invariant ("creator at 0, joiner at 1") is upheld by
  `RoomService.createRoom` and `joinRoom`; the mapper documents the
  contract in its JavaDoc.

**200 + `gameId: null` for waiting rooms, NOT a separate 204 / status.**

- Decision: a waiting room returns `200 OK` with `gameId: null` and
  `status: WAITING_FOR_PLAYER`.
- Alternatives: `204 No Content` for waiting rooms; a separate
  endpoint for "the game id has been minted".
- Why: simpler client contract. One endpoint, one shape, one parse
  path; the client branches on `gameId === null` (or `status`).
  Different status codes would force conditional parsing on the
  client; a separate endpoint would force two polls.

**404 reuses `ROOM_NOT_FOUND` from the 9-code allowlist; no expansion.**

- Decision: the 404 case uses the existing
  `RoomNotFoundException ‚Üí ROOM_NOT_FOUND` mapping that
  `POST /api/rooms/{id}/join` already produces.
- Alternatives: introduce a new code (e.g. `ROOM_GONE` for
  "expired").
- Why: the semantic is identical ‚ÄĒ the id does not name any room we
  know about. Feature 6.6 hardened the 9-code allowlist; adding a
  10th code without a new failure mode would defeat the
  consistency we paid for there.

**Broadcast outside the atomic block, with `try/catch + WARN log`.**

- Decision: `messagingTemplate.convertAndSend(...)` runs after
  `roomStore.compute(...)` returns, wrapped in `try/catch
  (RuntimeException)` that logs at `WARN` and does not rethrow.
- Alternatives: broadcast inside the lambda (atomic with the
  mutation); broadcast from the controller after the service
  returns.
- Why: same reasoning as `GameService.broadcastMoveEvent`. A
  broker-side throw inside the lambda would propagate out and look
  like a failed mutation ‚ÄĒ but the room and game are already in
  Redis. Doing the broadcast from the controller would split the
  service's responsibility across two layers. Outside-and-loud is
  the honest shape; the REST GET endpoint is the recovery path if
  the broadcast fails.

**`GameStore.findByRoomId` via SCAN, not a secondary index.**

- Decision: the GET endpoint resolves `gameId` by calling
  `GameStore.findByRoomId`, implemented in `RedisGameStore` as a
  `SCAN game:* + filter`.
- Alternatives: add a secondary index in Redis (`roomId ‚Üí
  gameId`); store `gameId` on the `Room` record.
- Why: the room-to-game relationship is one-to-(at-most-)one, the
  loop exits on the first hit, and the active-game cardinality is
  bounded by single-instance traffic. Adding a secondary index
  doubles every write path (insert + delete on terminal-game
  archive); adding `gameId` to `Room` leaks a cross-aggregate
  reference into the domain. `SCAN` is the lowest-coupling
  resolution.

## How this compares to what I know

- **STOMP topic `/topic/rooms/{roomId}` ‚Čą `fs2.concurrent.Topic[F,
  RoomEvent]`.** One publisher (the join service), N subscribers,
  fire-and-forget fan-out, no replay. The shape and the failure
  mode are identical. Spring's `SimpMessagingTemplate` is the
  side-effecting equivalent of `topic.publish1(event)`.
- **`sealed interface RoomEvent permits RoomJoinedEvent` ‚Čą Scala
  `sealed trait RoomEvent` with `final case class
  RoomJoinedEvent(...)`.** Same compile-time exhaustiveness, same
  closed hierarchy. Java's seals require the variants to be
  declared in the same module (or explicitly named in the
  `permits` clause); Scala's just require them in the same file.
- **Explicit `type` discriminator ‚Čą circe's
  `Configuration.withDiscriminator("type").withSnakeCaseConstructorNames`.**
  In Typelevel-land you would derive an `Encoder[RoomEvent]` that
  injects a `type` field with the variant name; here we set it by
  hand on the producer side and read it as a regular field on the
  consumer side. The visibility-in-source argument applies in both
  ecosystems ‚ÄĒ the discriminator-as-annotation pattern is just as
  invisible in Scala-with-derivation as it is in Java-with-Jackson.
- **`convertAndSend` ‚Čą `topic.publish1(event).unsafeRunSync()`.**
  Side-effecting publish at the edge of an effect-free domain
  block. Same separation we apply when keeping `IO` at the edges.
- **The mapper boundary ‚Čą a typeclass-derived `Encoder[Room,
  RoomDetailsResponse]`.** In Scala you would derive (or
  hand-write) an `Encoder` that pairs the domain shape with the
  wire shape; in Java the `@Component` mapper is the equivalent
  module-level singleton with one public method.
- **vs Node `socket.io`.** The Node version used `socket.emit` and
  `socket.to(room).emit` directly on the `Socket.IO` namespace,
  with implicit JSON. STOMP destinations are the same conceptual
  shape (server-managed subscription registry, broadcast by name)
  with a documented protocol on top. The discriminator pattern
  here is what you would build by hand in Node by adding an
  `event` field to every payload and switch-casing on it
  client-side ‚ÄĒ except the JS world rarely formalises the seal.

## Gotchas / things I learned the hard way

- **Spotless reformats long JavaDoc lines aggressively.** Multiple
  iterations of "run spotless, re-read, fix the next thing" before
  every file passed `spotless:check`. The fix is always
  `./mvnw spotless:apply` once and then re-read the file.
- **`GameStore.findByRoomId` was the only addition forced by this
  feature.** The original plan called for adding `findById` on the
  service if absent, but the harder problem was "given a roomId,
  find its game id" ‚ÄĒ the `Room` record doesn't carry it, so
  walking the game store by `roomId` was the simplest extension.
  `SCAN` over `KEYS` was the right Redis idiom (non-blocking,
  cursor-based, production-safe).
- **`MappingJackson2MessageConverter` ships with its own
  `ObjectMapper` that does not register `JavaTimeModule` ‚ÄĒ same
  trap as `GameWebSocketIT` had to work around for `MoveEvent`'s
  `Instant`.** `RoomJoinedEvent` does not currently carry an
  `Instant`, but I registered the module anyway in
  `RoomLifecycleIT` to keep the setup symmetric with
  `GameWebSocketIT` and to immunise the test against future fields.
- **Wire-format assertions need a `byte[]`-typed `StompFrameHandler`
  to see the JSON before the converter touches it.** Asking the
  converter for a `RoomJoinedEvent` would defeat the purpose of the
  discriminator test ‚ÄĒ we want to assert the field exists in the
  serialized output, not that the Java record has a getter for it.
