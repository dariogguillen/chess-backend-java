# Feature 23.93 — `GET /api/me/stats` (aggregate W/L/D record)

**Feature ID:** `me-stats` (from `feature_list.json`)

**Status:** done

---

## What we built

A single authenticated endpoint, `GET /api/me/stats`, that returns the caller's
overall chess record: `{ total, wins, losses, draws, unknown, winRate }`. It is
built directly on the `games.result` column shipped by feature 23.92
(`game-result-persistence`) — the win/loss split cross-references each game's
result with the user's side (`white_user_id` vs `black_user_id`), the buckets
reconcile (`total == wins + losses + draws + unknown`), and the whole thing is
computed by one JPQL conditional-sum query that never loads the game rows. There
is no Flyway migration and no new error code.

## Java / Spring concepts that appear

- **JPQL constructor projection with conditional `SUM(CASE …)`.** The aggregate
  is a single `@Query` on `GameHistoryRepository.statsForUser` that does
  `SELECT new …UserGameStatsView(COUNT(g), COALESCE(SUM(CASE WHEN … THEN 1L ELSE
  0L END), 0L), …)`. JPQL's `SELECT new <fqn>(…)` invokes the record's canonical
  constructor at result-set extraction time, so the five aggregate columns come
  back as one immutable `UserGameStatsView` — no entity is materialised, no row
  is loaded. The component order of the record **must** match the SELECT list
  exactly; Hibernate binds positionally and a mismatch is a silent runtime bug,
  not a compile error. See the Spring Data JPA reference on
  [constructor expressions](https://docs.spring.io/spring-data/jpa/reference/jpa/query-methods.html).
- **`COALESCE` over an empty `SUM`.** `SUM` over zero matching rows returns
  `NULL` in SQL, not `0`. For a user with zero games the `WHERE` still yields a
  one-row aggregate (`COUNT = 0`, every `SUM = NULL`); binding `null` into the
  `long` record components would throw at extraction. Wrapping every sum in
  `COALESCE(SUM(…), 0L)` removes the only `null` source, so the primitive
  `long` binding is safe and the empty case returns `total = 0` with all-zero
  buckets. The `MeStatsIT` empty-history case proves this.
- **`@Transactional(readOnly = true)`.** `GameHistoryService.statsFor` is a
  read-only transaction wrapping a single query. Spring wraps the bean in a
  proxy that opens a read-only transaction (a hint the JDBC driver and Hibernate
  can use to skip dirty-checking / flush) around the method. See the
  [Spring `@Transactional` docs](https://docs.spring.io/spring-framework/reference/data-access/transaction/declarative/annotations.html).
- **`@AuthenticationPrincipal` + the security chain.** `MeStatsController.stats`
  takes `@AuthenticationPrincipal User user` and reads `user.getId()`. The
  `/api/me/**` surface is covered by `SecurityConfig`'s
  `anyRequest().authenticated()` rule, so an unauthenticated call is 401'd by the
  filter chain before the controller body runs — the principal is non-null by
  construction. No per-endpoint security change was needed.

## Decisions taken

- **Decision:** compute the aggregate in one SQL query (conditional sums), not by
  loading the user's games and folding in memory.
  - **Alternatives considered:** reuse `findByUserId` (already paginated) and sum
    in Java; or add a derived-count repository method per bucket (four queries).
  - **Why this one:** the counts are a textbook aggregate — the database does it
    in a single index-backed row scan with no row materialisation, which is both
    correct under pagination (folding a page would only count one page) and
    cheaper than shipping every row to the app just to discard all but the count.
    Four separate count queries would be four round-trips for data that is one
    `GROUP`-less aggregate.
- **Decision:** `winRate` is computed in Java (`MyStatsResponse.from`), not in
  JPQL.
  - **Alternatives considered:** a `CAST(... AS double) / NULLIF(decided, 0)`
    expression inside the SELECT.
  - **Why this one:** the divide-by-zero guard (return `0.0` when no games are
    decided) is trivial and unambiguous in Java; pushing it into JPQL means a
    `NULLIF` plus a `COALESCE` plus a cast, which is harder to read and binds a
    `null`-able `Double` we'd have to unwrap anyway. The plan called this out
    explicitly.
- **Decision:** legacy `NULL`-result rows surface as a separate `unknown`
  bucket, counted in `total` but excluded from W/L/D and from the `winRate`
  denominator.
  - **Alternatives considered:** silently exclude them from `total` entirely.
  - **Why this one:** `total` then reconciles with the `/api/me/games` list the
    frontend already shows (both count the same rows), and the `unknown` count is
    honest about old `ABANDONED` games whose winner 23.92 could not recover from
    the final FEN. Excluding them from `total` would make the two surfaces
    disagree on how many games the user has played.
- **Decision:** the service returns the web DTO `MyStatsResponse` directly.
  - **Why this one:** it matches the established house pattern — `ProfileService`
    already returns `MeResponse` (a `web.auth` DTO). The view-to-response mapping
    (and the win-rate computation) lives in one place, and the controller stays a
    one-line delegation.

## How this compares to what I know

- **In Scala / Typelevel this would be…** the same choice you'd make with doobie
  or Quill: a single `sql"… COUNT(*), SUM(CASE WHEN …)"` aggregate decoded into a
  case class via `.query[UserGameStats]`, rather than `streamGames.compile.fold`
  over the rows in `F[_]`. The `COALESCE(SUM(…), 0L)` is the SQL analogue of the
  empty-`fold` problem: an empty stream folds to your zero, but `SUM` over no
  rows is `NULL`, so you push the zero into SQL. The `winRate` guard is exactly
  what you'd write as `if (decided == 0) 0.0 else wins.toDouble / decided` — a
  pure function over the decoded row, kept out of the query for the same
  readability reason. And `JPQL SELECT new Foo(...)` is doobie's `.query[Foo]`
  derivation: positional column-to-field binding, with the same footgun that the
  field order must match the SELECT.
- **In Node this would be…** a single parameterised aggregate (`SELECT count(*),
  sum(case when … then 1 else 0 end) …`) via `pg` / Knex / Prisma's
  `$queryRaw`, then a tiny mapper that reads the row and computes `winRate`,
  guarding the divide-by-zero in JS. Same shape, same trade-off.

## Gotchas / things I learned the hard way

- The `COALESCE(…, 0L)` is not optional cosmetics: drop it and the *only* failing
  case is the empty-history user, which is easy to miss if you only test users
  who have games. The empty-history IT case is the canary.
- Spotless reflows long Javadoc lines on `./mvnw verify`; running
  `./mvnw spotless:apply` after writing the heavily-documented repository query
  and view record is faster than guessing the wrap points by hand.
