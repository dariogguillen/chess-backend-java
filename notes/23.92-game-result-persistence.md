# Feature 23.92 — Persist the game result (who won)

**Feature ID:** `game-result-persistence` (from `feature_list.json`)

**Status:** done

---

## What we built

Until now the archived `games` table stored only `status` — *how* a game ended
(`CHECKMATE`, `STALEMATE`, `DRAW`, `ABANDONED`, `TIMEOUT`) — but never *who*
won. The winner was computed at runtime in all four terminal paths and broadcast
on the STOMP events, then thrown away at archive time. This feature introduces a
`GameResult` enum `{ WHITE_WIN, BLACK_WIN, DRAW }`, threads it through the `Game`
domain record so it is set at every terminal transition, persists it via a new
nullable `games.result` column (V4 migration + a `final_fen` backfill of existing
rows), and exposes it on both game-history surfaces. It is the foundation for the
next feature, `me-stats`, which aggregates per-user W/L/D.

## Java / Spring concepts that appear

- **Backwards-compatible record evolution with a nullable trailing component.**
  Java records have one canonical constructor whose parameter list *is* the
  component list. Adding a 14th component (`result`) to `Game` would break every
  `new Game(...)` call site and every Jackson deserialisation of a Redis snapshot
  written before the field existed. The established pattern in this codebase
  (used already for `timeControl` and `botElo`) is to add the component to the
  canonical constructor but keep the *older arities* as explicit convenience
  constructors that delegate with the new component defaulted to `null`. So
  `new Game(8 args)`, `new Game(12 args)`, and `new Game(13 args)` all still
  compile and all pass `result = null`. Jackson, deserialising an old snapshot
  that lacks the `result` JSON property, fills it with `null` through the
  canonical constructor — no migration of in-flight Redis data needed. See
  `domain/Game.java`.
- **`@Enumerated(EnumType.STRING)`** — JPA maps the `GameResult` enum to a
  `VARCHAR(20)` column storing the constant's `name()` (`"WHITE_WIN"`), not its
  ordinal. STRING is forward-compatible: inserting a new enum constant in the
  middle of the declaration later cannot silently corrupt existing rows the way
  `ORDINAL` would. `result` is mapped exactly like the existing `status` field.
  See `persistence/GameEntity.java`.
- **JPQL constructor expressions (`SELECT new ...`).** The history queries don't
  return entities; they project straight into an immutable record
  (`ArchivedGamePlayerView`) via `SELECT new <fqn>(g.id, ..., g.result, ...)`.
  The SELECT-list order must match the record's canonical constructor posit-for-
  position, so adding `result` meant inserting `g.result` at the same index in
  both queries and in the record. This projection-at-query-time matters because
  the app runs with `spring.jpa.open-in-view: false` — outside the service
  transaction the entity manager is closed and lazy associations are unreachable.
  See `persistence/GameHistoryRepository.java`.
- **Flyway versioned migration + data backfill.** `V4__add_game_result.sql` is
  two statements: an `ALTER TABLE ... ADD COLUMN` and a one-time `UPDATE` that
  derives the historical result from `status` + `final_fen`. Postgres'
  `split_part(final_fen, ' ', 2)` extracts the FEN active-color field (`'w'` /
  `'b'`) to decide the winner of old `CHECKMATE` / `TIMEOUT` rows. `ddl-auto:
  validate` then checks the new column against the entity mapping at boot.

## Decisions taken

- **Decision:** one shared winner-derivation in `GameResult`, not four copies.
  - **Alternatives considered:** leave the white-vs-black ternary inline in each
    of the four terminal paths (the status quo for the broadcast `winnerId`).
  - **Why this one:** the same `whiteToMove ? ... : ...` logic appeared in
    `GameService.applyMove`, `GameAbandonService`, `GameTimeoutService`, and
    `BotMoveService`. Centralising it as `GameResult.fromWinner(game, winnerId)`
    and `GameResult.fromLoserToMove(whiteToMove)` means the persisted result and
    the broadcast `winnerId` are derived from the *same* function — they cannot
    drift. Two helpers because two call shapes exist: the abandon/bot paths know
    the *winner id* directly; the checkmate/clock paths know the *side to move*
    (the loser).
- **Decision:** stamp `result` *inside* the `gameStore.compute` lambda on the
  three async paths (alongside `withStatus`), then archive outside the lambda.
  - **Alternatives considered:** set `result` only on the object passed to
    `archive(...)`, leaving the Redis active copy without it.
  - **Why this one:** the plan explicitly required that the Redis active copy and
    the archived Postgres row not disagree on the winner. Stamping inside the
    lambda makes the Redis write carry the result too; the archive still runs
    outside the lock (terminal status is irrevocable, so there's no further-move
    race, and the JDBC round-trip stays out of the critical section).
- **Decision:** persist the derived result at *write* time rather than recompute
  it on *read*.
  - **Alternatives considered:** store nothing new and derive W/L/D on every read
    from `status` + `final_fen`.
  - **Why this one:** the read path stays a trivial column read, and `me-stats`
    can aggregate with a plain `GROUP BY result` instead of re-parsing FENs in
    SQL on every stats request. The backfill proves the read-time derivation is
    *possible*; persisting makes it *cheap and reusable*.
- **Decision:** leave legacy `ABANDONED` rows' `result` as `NULL`.
  - **Why this one:** the abandoner (the loser) is the disconnected player — an
    identity known only at runtime when the grace timer fired, never encoded in
    the FEN. It is genuinely unrecoverable for old rows, so `NULL` ("unknown") is
    the honest value. New `ABANDONED` games archived after this feature *do*
    carry the result. The wire DTOs expose `result` as nullable so the frontend
    renders these legacy unknowns gracefully.

## How this compares to what I know

- **In Scala / Typelevel this would be...** The `Game` record is a `case class`,
  and the `withResult` / `withStatus` mutators are exactly `.copy(result = ...)`.
  The big difference: a Scala `case class` gets `.copy` for free with named/
  default args, so backwards-compatible field addition is trivial — add
  `result: Option[GameResult] = None` and every existing `.copy` / construction
  with named args keeps working. Java records have no `.copy` and no default
  parameters, so we hand-write the `withX` methods *and* the overloaded
  convenience constructors to simulate "older arity still compiles". `GameResult`
  itself is a closed ADT — in Scala a `sealed trait GameResult` with three
  `case object`s; the `switch` over `GameStatus` in `deriveResult` is a pattern
  match, and Java's arrow-`switch` over an enum is the same exhaustive-ish shape
  (with a `default` arm for the non-terminal statuses).
- **In Node this would be...** `GameResult` is a string-union type
  `'WHITE_WIN' | 'BLACK_WIN' | 'DRAW'`, and the column is just a nullable text
  field — no enum machinery, the discipline lives entirely in TypeScript types
  and is gone at runtime. The "persist-derived-at-write vs recompute-on-read"
  trade-off is identical in any stack; here JPA's `@Enumerated(STRING)` gives us
  a runtime guarantee (the only writer can't produce an off-alphabet value) that
  a Node + raw-SQL stack would have to enforce with a `CHECK` constraint or trust
  the app layer for — the same reasoning the codebase already applies to
  `status`.

## Gotchas / things I learned the hard way

- The two derivation helpers are *not* the same mapping and it's easy to invert
  one. `fromLoserToMove(whiteToMove)` is used where `whiteToMove` is the side
  *to move* at the terminal position (the mated/flagged loser → the *other* side
  wins). But in `GameService.applyMove` the relevant `whiteToMove` is the side
  that *just moved* (the mate-deliverer → the winner), so that path uses its own
  inverted `deriveResult`, not `fromLoserToMove`. Same boolean name, opposite
  meaning.
- Seeding rows via raw `JdbcTemplate` in the backfill IT failed with "Can't infer
  the SQL type to use for an instance of `java.time.Instant`" — the Postgres JDBC
  driver won't bind a bare `Instant`. Wrapping it in `Timestamp.from(instant)`
  fixed it. (The production path never hits this because Hibernate handles the
  `Instant ↔ TIMESTAMPTZ` binding; it only bit the hand-rolled JDBC test.)
