# Feature 23.7 — Lichess-style bot strength via Fairy-Stockfish

**Feature ID:** `bot-strength-fairy-stockfish` (from `feature_list.json`)

**Status:** in progress

---

## What we built

Feature 23.5 limited the bot's strength with official Stockfish's
`UCI_LimitStrength` + `UCI_Elo`, which **floors at ~1320 Elo** — it
physically cannot play like a beginner. This feature adopts **Lichess's
model** instead: it switches the engine to **Fairy-Stockfish** (whose
`Skill Level` option spans `-20..20`, and the negative levels are what
reach sub-1320 play) and replaces the `UCI_Elo` strength model with a
pure mapping from the requested **Elo** to a `(Skill Level, depth)` pair,
interpolating Lichess's published 8-tier table. The public interface
stays **Elo** end to end (`CreateRoomRequest.botElo`, `Game.botElo`); only
the internal mapping and the emitted UCI options change, and the accepted
range widens down to ~400.

## Java / Spring concepts that appear

- **A pure mapping as a `final class` + static method (`BotEloMapping`).**
  The strength model is branchy, pure arithmetic with no Spring
  dependency, so it is a stateless utility (private constructor, one
  static `forElo`) — the same shape as `UciMove`. It is the single source
  of truth and is unit-tested directly, no context boot. This is the Java
  idiom for "a function that happens to need a home"; there is no
  top-level `def` to hang it on.
- **A package-private seam for testing process I/O without the process
  (`FairyStockfishBotEngine.buildSearchCommands`).** Spawning a real
  subprocess in a unit test is slow and needs the binary. Factoring the
  UCI command *construction* into a package-private method lets a
  same-package test (`FairyStockfishBotEngineTest`) assert the exact lines
  (`Skill Level` + `go depth`, and the *absence* of `UCI_Elo`) with zero
  I/O. Java's package-private visibility is the access level that makes a
  seam testable without making it public API.
- **`ProcessBuilder` + guaranteed teardown (`destroyForcibly` in
  `finally`).** Unchanged from feature 23, but worth restating: the
  subprocess is always force-killed in a `finally`, and the
  `process.waitFor(timeout, MILLISECONDS)` gives a hard wall-clock
  backstop. This is the JVM's manual equivalent of a bracketed resource.
- **`@ConfigurationProperties` record with a validating compact
  constructor (`BotProperties`).** Removing the `moveTime` component and
  re-pointing `enginePath` happens here; the compact constructor still
  fails the boot fast on a bad `default-elo`. Constants
  `BotEngine.MIN_BOT_ELO` / `MAX_BOT_ELO` are referenced by both the DTO
  `@Min`/`@Max` *and* this validation, so the request bound and the
  configured default can never drift — a compile-time single source of
  truth.
- **Pinning a container binary with checksum verification (`Dockerfile`,
  `ADD --chmod` + `sha256sum -c`).** Fairy-Stockfish is not in apt, so the
  runtime stage downloads a pinned GitHub release asset and verifies its
  SHA-256. `ADD` with a URL fetches at build time; `--chmod=755` makes it
  executable in one layer; the `sha256sum -c` line fails the build if the
  bytes ever change. Reproducible builds without an apt repo.

## Decisions taken

- **Decision: switch to Fairy-Stockfish; keep the Elo interface; map Elo →
  `(Skill Level, depth)`.**
  - **Alternatives:** (a) keep official Stockfish and expose `Skill Level`
    directly to the API; (b) a hybrid that uses `UCI_Elo` above 1320 and
    `Skill Level` below it.
  - **Why this one:** an engine cannot be made to "play at 850 Elo" by one
    knob. `UCI_Elo` has a hard ~1320 floor; `Skill Level` alone is an
    opaque `-20..20` dial users do not think in. Lichess's answer — one
    engine (Fairy-Stockfish), two knobs together (`Skill Level` + capped
    `depth`), and a public **Elo** that maps onto them — is a coherent,
    battle-tested model. A hybrid would mean two strength regimes with a
    discontinuity at 1320; the single Fairy model is simpler and what
    Lichess actually ships.

- **Decision: rename `StockfishBotEngine` → `FairyStockfishBotEngine`.**
  - **Alternatives:** keep the old class name.
  - **Why:** the adapter no longer drives official Stockfish, and the
    distinction (Fairy's extended `Skill Level` range) is the whole point
    of the feature. The class name should not lie. (The bot *player's*
    `displayName` stays `"Stockfish"` — that is a user-facing label and a
    cross-repo string, out of scope for an engine swap.)

- **Decision: interpolate the tiers instead of snapping to 8 buckets.**
  - **Alternatives:** snap the requested Elo to the nearest of Lichess's 8
    tiers (what Lichess itself does — it only offers 8 levels).
  - **Why:** our API already takes a continuous Elo. Interpolating keeps
    850 distinct from 1500 rather than collapsing a wide range into 8
    values, for a small pure function. The anchors and the floor/ceiling
    clamp are exactly Lichess's; only the in-between is ours.

- **Decision: `Use NNUE false` (classical eval).**
  - **Alternatives:** ship Fairy-Stockfish's NNUE net file in the image
    and run with NNUE on.
  - **Why:** NNUE needs a multi-megabyte net file alongside the binary,
    and the `Skill Level` weakening works identically under classical
    eval. For portfolio-scale bot play, classical is plenty and the image
    stays small (just the ~2.3 MB binary, no net).

- **Decision: `move-timeout` default = 30s; drop `move-time`.**
  - **Why:** the search is now bounded by `go depth <d>`, not wall-clock,
    so `move-time` is meaningless. The hard `move-timeout` is now purely a
    backstop kill. I measured `go depth 22` (the top tier) at ~400 ms
    single-core with classical eval on the dev box; 30s is a generous
    margin for a slower t3.micro.

- **Decision: pin tag `fairy_sf_14`, asset `fairy-stockfish_x86-64`.**
  - **Alternatives:** the latest tag (`fairy_sf_14_0_1_xq`) only ships
    *largeboard* assets; the `-bmi2` / `-modern` builds are CPU-specific.
  - **Why:** `fairy_sf_14` has the chess-only (non-largeboard) generic
    `x86-64` build — chess is all we need, and the plain `x86-64` target
    (not `-bmi2`/`-modern`) runs on any x86-64 CPU including the t3.micro.
    SHA-256 `ab6b8582…ee150f`, pinned and verified in the Dockerfile.

## Verifying the mapping against lila (the authoritative source)

The plan's table was a forum summary; the directive was to verify it
against lila. Lichess's play-with-the-computer searches are actually run
by the Rust **fishnet** worker, so the authoritative `(skill, depth)`
table is the `SkillLevel` enum in
[`lichess-org/fishnet`, `src/api.rs`](https://github.com/lichess-org/fishnet/blob/master/src/api.rs)
(`skill_level()` and `depth()` methods):

| Tier | Skill | Depth | ~Elo |
|------|-------|-------|------|
| 1 | -9 | 5 | <400 |
| 2 | -5 | 5 | 500 |
| 3 | -1 | 5 | 800 |
| 4 | 3 | 5 | 1100 |
| 5 | 7 | 5 | 1500 |
| 6 | 11 | 8 | 1900 |
| 7 | **16** | 13 | 2300 |
| 8 | 20 | 22 | 2800+ |

**Discrepancy found:** the plan's forum-derived table had tier 7 skill =
**15**; lila's actual source says **16**. We follow lila (16). Everything
else matched. The approximate Elos are Lichess's own published estimates
(the same numbers surface across the lila forum threads); the last anchor
is pinned at our `MAX_BOT_ELO` (3190) rather than the published ~2800 so
the documented ceiling maps cleanly to full strength.

## How this compares to what I know

- **In Scala / Typelevel this would be...** `BotEloMapping.forElo` is a
  total function `Int => EngineStrength` — exactly a `case class
  EngineStrength(skill: Int, depth: Int)` plus a lookup-table-with-linear-
  interpolation function. And the reference implementation *is* Scala:
  lila is a Typelevel/Play codebase, so the original `SkillLevel` notion
  lived in Scala before the search worker was rewritten in Rust. Our
  `BotEloMapping` is a small port of that idea — the same anchors, made
  continuous. The `final class` + `private constructor` + static method
  dance is just Java's way of writing what Scala would put in an `object`.
- **The seam (`buildSearchCommands`) in Typelevel terms** is what you'd do
  by extracting a pure `List[String]` builder out of an `fs2`/`cats-effect`
  process stream so you can test the protocol without `Resource[F,
  Process]`. Java has no effect type here, so the "pure part" is just a
  method returning `List<String>` and the "effectful part" is the
  `ProcessBuilder` loop; the test targets the pure part.
- **In Node this would be...** `child_process.spawn(enginePath)` writing
  UCI lines to `stdin` and parsing `stdout` line-by-line — the same
  spawn-per-move shape. The `(skill, depth)` map is a plain function; the
  "pin a binary + verify checksum in the Dockerfile" step is identical
  (you'd `ADD` the release asset and `sha256sum -c` regardless of runtime).

## Gotchas / things I learned the hard way

- The "current" Fairy-Stockfish release tag shipped *only* largeboard
  assets; I had to step back to `fairy_sf_14` to get the chess-only
  `x86-64` build. Always check the actual asset list, not just "latest".
- The authoritative skill/depth table is in the **Rust** fishnet worker,
  not in lila's Scala `modules/fishnet` (which dispatches work but doesn't
  hold the numbers anymore). The forum table was *almost* right (tier 7
  off by one). Verifying against the real source caught it.
- `go depth 22` is fast (~400 ms) — depth, not time, is the right bound
  for a *consistent* strength tier; a `movetime` bound would make the same
  tier stronger on a faster CPU.

## Follow-ups (documented, not in scope)

- **Phase 2: MultiPV-4 + randomized weakness.** Lichess additionally runs
  a MultiPV-4 search and adds a `Skill Level`-derived random weakness to
  each line's score before picking the best — the most expensive part of
  its model, for more human-feeling blunders. Phase 1 (this feature) uses
  Fairy's *native* `Skill Level` randomization, which already reaches
  credible low Elo without MultiPV. Deferred.
- **Surfacing the played strength in game history** (e.g. "vs the bot
  (1500)") — the Elo lives only on the active Redis `Game`, not the
  Postgres archive. Pre-existing follow-up from 23.5, unchanged.
