# Feature 23.91 — Profile editing (expose createdAt, rename, change password)

**Feature ID:** `profile-edit` (from `feature_list.json`)

**Status:** done

---

## What we built

Three additive changes to the authenticated user's own-profile surface, with
**no Flyway migration** (the `created_at`, `display_name`, and `password_hash`
columns already existed): (1) `GET /api/me` now returns `createdAt` ("member
since"); (2) `PATCH /api/me` renames the display name; (3) `PUT
/api/me/password` changes the password after verifying the current one. This
rounds out the editable own-profile so the frontend's profile page can show the
account age and let the user edit their identity.

## Java / Spring concepts that appear

- **Static factory on a record (`MeResponse.of(User)`)** — a record's
  canonical constructor is its primary API, but nothing stops you adding static
  factory methods. We routed all three construction sites (`MeController`,
  `AuthService`, `ProfileService`) through one factory so the `createdAt`
  addition landed in a single place instead of three. Records can declare
  static members freely; only instance state is restricted to the components.
- **JPA dirty checking on a managed entity** — `@Transactional` opens a
  persistence context (Hibernate `Session`); any entity loaded *inside* that
  context is "managed", and Hibernate tracks field changes and flushes an
  `UPDATE` at commit automatically. `ProfileService` re-loads the `User` via
  `userRepository.findById(...)` inside the transaction, mutates it through
  `User.rename` / `User.changePasswordHash`, and returns — no explicit `save`
  needed. See the
  [Hibernate persistence-context docs](https://docs.jboss.org/hibernate/orm/6.4/userguide/html_single/Hibernate_User_Guide.html#pc).
- **Detached vs managed entity** — the `@AuthenticationPrincipal User` the
  controller resolves was loaded by the JWT filter on a *now-closed* persistence
  context, so it is **detached**: mutating it directly would not persist. This
  is exactly why the service re-loads by id rather than mutating the principal.
  We pass only `user.getId()` from the controller to the service to make that
  boundary obvious.
- **Rich domain mutators vs exposed setters** — `User` keeps its setters
  package-private (JPA-internal) and exposes `rename(String)` /
  `changePasswordHash(String)` as the public, intention-revealing mutation API.
  A profile edit reads as a domain operation, not a raw `setDisplayName`.
- **`PasswordEncoder.matches` against a null hash** — Spring's
  `BCryptPasswordEncoder.matches(raw, null)` returns `false` (it does not throw).
  That single fact is what makes the OAuth-only-account case fall out for free:
  no special branch, no leak of whether the account has a password.
- **`@ResponseStatus(HttpStatus.NO_CONTENT)` + `void`** — the password endpoint
  returns 204 with no body. A `void` controller method plus `@ResponseStatus`
  is the idiomatic way to say "succeeded, nothing to return".
- **`@PatchMapping` / `@PutMapping`** — PATCH for the partial profile update
  (only `displayName`), PUT for the password replacement (a full replace of the
  resource's password sub-state). Both resolve via the same request-mapping
  machinery as `@GetMapping`.

## Decisions taken

- **Decision:** Two endpoints (`PATCH /api/me` + `PUT /api/me/password`) rather
  than one `PATCH` with optional fields.
  **Alternatives considered:** a single `PATCH /api/me` taking optional
  `displayName` / `currentPassword` / `newPassword`.
  **Why this one:** the two operations have different bodies, different success
  codes (200-with-body vs 204), and different security posture (the password
  change must verify the current password). A combined endpoint would need
  conditional validation and a branchy handler; two focused endpoints are
  clearer and each documents one contract.

- **Decision:** A new `ProfileService` rather than extending `AuthService`.
  **Alternatives considered:** adding `updateDisplayName` / `changePassword` to
  `AuthService`.
  **Why this one:** `AuthService` is about register/login (issuing tokens); the
  profile mutations share neither flow nor failure semantics with
  authentication. Keeping them apart keeps each service a single use case.

- **Decision:** OAuth-only accounts (null `password_hash`) get the same `401
  INVALID_CREDENTIALS` as a wrong current password — no new error code.
  **Alternatives considered:** a dedicated `OAUTH_ONLY_ACCOUNT` 409/422.
  **Why this one:** distinguishing the two would leak whether an account is
  OAuth-only. `matches(current, null) == false` already produces the uniform
  401, so the secure behaviour is also the simplest. The `ErrorResponse`
  allowable-values canary stays at 21 codes.

- **Decision:** No JWT re-issue on either change.
  **Why this one:** a rename touches no JWT claim (`sub` = userId, the only
  identity claim is `email`), so existing tokens stay valid. A password change
  *cannot* revoke stateless tokens — there is no server-side token store — so
  the old token remains valid until expiry. Documented as the accepted
  trade-off of the stateless-JWT model.

- **Decision:** A missing-by-id user in the service throws `IllegalStateException`
  (→ 500), not a new client-facing error code.
  **Why this one:** the JWT filter has already loaded this user for the request
  to be authenticated, so a missing-by-id here is an impossible state, not a
  client error. There is no existing `UserNotFoundException` in the hierarchy,
  and inventing a 4xx code for an unreachable branch would be wrong (it is not a
  client mistake). A 500 is the honest signal.

## How this compares to what I know

- **In Scala / Typelevel this would be...**
  - The **rich domain mutators** map to keeping the case class / state private
    and exposing intention-named transitions. With an immutable model you would
    return a *new* `User` from `rename` rather than mutate in place — JPA's
    managed-entity model is the inversion of that: you mutate and the framework
    diffs and flushes.
  - **JPA dirty checking** is the big divergence from doobie. In doobie you
    write the `UPDATE` explicitly: `sql"update users set display_name=$name
    where id=$id".update.run.transact(xa)`. There is no change-tracking; the
    SQL is the source of truth. Here, loading the row inside a `@Transactional`
    method and mutating a field *is* the update — Hibernate emits the SQL at
    commit. Less boilerplate, but you must understand the
    managed/detached boundary or the change silently no-ops (which is precisely
    the trap the detached `@AuthenticationPrincipal` would spring).
  - **`PasswordEncoder.matches` against a null hash** — in tsec you would have
    a `PasswordHash[BCrypt]` that is non-null by type; modelling "no password"
    as `Option[PasswordHash]` would force you to handle the `None` case
    explicitly at the type level. Java's `String passwordHash` being nullable
    pushes that case to a runtime `false` from `matches`, which here happens to
    be exactly the behaviour we want, but the type does not warn you about it.
  - **PATCH partial-update semantics** — http4s would have you decode the body
    and pattern-match on which fields are present. We dodged that entirely by
    splitting into two endpoints, so each has a total (non-optional) body.

- **In Node this would be...** an Express `PATCH /api/me` and `PUT
  /api/me/password` reading `req.user.id` from the JWT middleware, then a Prisma
  `prisma.user.update({ where: { id }, data: { displayName } })`. Prisma, like
  doobie, makes the write explicit — closer to doobie than to JPA's implicit
  dirty-check flush. The `bcrypt.compare(current, hash)` returns `false` for a
  null hash much like Spring's `matches`.

## Gotchas / things I learned the hard way

- The detached-principal trap is real: had `ProfileService` mutated the
  `@AuthenticationPrincipal User` directly instead of re-loading by id, the
  change would have silently not persisted (the principal is bound to a closed
  context). Passing only `user.getId()` to the service makes the boundary
  impossible to get wrong.
- `currentPassword` deliberately carries only `@NotBlank`, not `@Size` — a
  wrong-but-well-formed current password must be a 401, not a 400, so the
  strength bounds live only on `newPassword`. Mirrors the register/login
  password-policy-leak-avoidance choice from feature 17.
