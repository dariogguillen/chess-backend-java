## Feature 11.8 — INFO logs on successful STOMP broadcasts

**Feature ID:** `broadcast-observability` (from `feature_list.json`)

**Status:** in progress

---

## What we built

An operational-logging gap closed at the six STOMP broadcast sites:
every `messagingTemplate.convertAndSend(...)` now emits a single
INFO line on success, carrying the destination and the key payload
identifiers. No behavioural change — no endpoint, no DTO, no STOMP
contract is touched, and the existing WARN-on-`RuntimeException`
catches stay exactly as they were. The test suite stays at 181 tests
with **zero** modifications under `src/test/`.

## Java / Spring concepts that appear

- **`SLF4J` log levels as an operational contract.** `org.slf4j.Logger`
  exposes `info`, `warn`, `error`, `debug`, `trace`. In Spring Boot
  the default Logback config emits everything at `INFO` and above to
  stdout, while `DEBUG`/`TRACE` are gated behind explicit
  `logging.level.<package>=DEBUG` overrides. Choosing `INFO` here is
  a deliberate operational decision: the broadcast confirmation must
  be visible **without** any operator opt-in. `DEBUG` would recreate
  the original gap — operator has to know which package to enable
  before they can answer "did the broadcast actually go out?". See
  [SLF4J's manual on log levels](https://www.slf4j.org/manual.html)
  and [Spring Boot's logging reference](https://docs.spring.io/spring-boot/reference/features/logging.html).
- **Operational logging vs developer logging.** A developer log
  (`DEBUG`/`TRACE`) is for the person currently editing the code
  with a debugger close by. An operational log (`INFO`/`WARN`/`ERROR`)
  is for whoever is **reading the production stream weeks from now**
  trying to answer "what did the server do at 14:32?". The two
  audiences want different signal-to-noise ratios, and that ratio is
  encoded in the level. The bug investigated in feature 11.7 (see
  below) crossed that line: the question was operational ("did the
  broadcast happen?") and the answer required developer-level
  tooling (TRACE on three Spring packages). This feature realigns
  the levels with the audience.
- **Operational logs sit outside the behavioural-test boundary.** The
  behavioural contract of `GameService.applyMove` is "the game
  state mutates and a `MoveEvent` is published to
  `/topic/games/{gameId}`" — both already covered by the
  `STOMP IT`-shaped tests. The log line that describes that publish
  is *observability about the contract*, not the contract itself. If
  a test asserted `"Broadcasted MoveEvent to ..."`, the log shape
  would become a behavioural contract, and any future log refactor
  would ripple into the test suite. The discipline is to keep log
  assertions out of behavioural tests and exercise the shape only via
  the explicit Logback test setup (we have none today, and that is
  fine).

## Decisions taken

**INFO level, not DEBUG.**

- Decision: every successful broadcast logs at `INFO`.
- Alternatives: `DEBUG`, gated behind
  `logging.level.io.github.dariogguillen.chess=DEBUG`.
- Why: the case study in feature 11.7 just proved these messages
  have real operational value — the next operator should not need
  to know which package to enable before they can answer "did the
  broadcast actually go out?". `DEBUG` would push the burden back
  onto the operator and recreate the exact gap this feature closes.
  The volume risk is bounded (one log per broadcast, broadcasts are
  inherently event-driven — there is no inner loop emitting these).

**After `convertAndSend`, not before.**

- Decision: the INFO line lives **inside** the existing `try` block,
  on the line immediately after `convertAndSend(...)` returns.
- Alternatives: emit a "Broadcasting..." log **before** the call, or
  emit both around it.
- Why: the WARN-on-`RuntimeException` catch already documents the
  "entered the method but failed" state. The "after" log specifically
  confirms `convertAndSend` returned without throwing — which is the
  state nothing logs today. The combined picture is now complete:
  WARN on failure, INFO on success, no third state to enumerate. A
  "before" log would be redundant with the WARN (the WARN's mere
  presence implies entry) and double the volume on every successful
  broadcast.

**No shared helper, inline at each site.**

- Decision: six explicit `log.info(...)` calls at the six broadcast
  sites, with per-event identifiers.
- Alternatives: a `broadcastWithLog(destination, payload, identifiers)`
  helper extracted into a shared component, taking a
  `Map<String, Object>` of identifiers.
- Why: each site logs a different shape — `gameId`/`joinerId` here,
  `movedBy`/`status` there, `playerId`/`side`/`gracePeriodEndsAt`
  somewhere else, `count` at the viewer-count site. A shared helper
  would either lose the typed identifiers (everything-as-`Map`,
  losing type-safety) or proliferate into one method per shape
  (no de-duplication gained). Six lines of explicit code beat one
  badly-fitting abstraction; this matches the "we deliberately did
  *not* extract a shared archive helper" decision documented in
  `notes/11-disconnect-handling.md` for the same reason.

**Identifiers in the log, payload omitted.**

- Decision: the INFO line carries identifying keys (UUIDs, short
  codes, sides, counts, deadlines) but **not** the full payload
  object.
- Alternatives: log the whole event (`event.toString()` via the
  record's auto-generated `toString`).
- Why: payloads can be large — a `MoveEvent` includes the full FEN
  after the move, a `RoomJoinedEvent` carries the joiner record. At
  `INFO` the priority is "operator can grep by id and confirm the
  event happened", not "operator can reconstruct the wire payload".
  Anyone needing the full payload still has TRACE on
  `org.springframework.messaging.simp.broker` available as an
  operator-time tool. The identifier set chosen per site is the
  minimum needed to cross-reference the broadcast against the
  upstream mutation log (room creation, move applied, abandon, etc.).

**No tests.**

- Decision: this feature adds **zero** lines under `src/test/`.
- Alternatives: an assertion-based test using a Logback list-appender
  to capture `INFO` records and pattern-match the message.
- Why: operational logs are not behavioural contract (see the
  "outside the behavioural-test boundary" concept above). Coupling a
  test to the log shape would freeze the wording on disk and convert
  a future "let's rename the field for clarity" refactor into a
  test-suite change. The 181 existing tests stay green as the
  regression signal — operational logs cannot affect behavioural
  outcomes, so any test churn would itself be a red flag indicating
  the change accidentally altered behaviour.

## The case study — feature 11.7's bug #3 investigation

During feature 11.7's investigation the question "is the
`RoomJoinedEvent` actually being broadcast?" arose. The codebase
logged WARN only on `RuntimeException`, so a silent successful path
was indistinguishable from a path that never ran. Confirming the
broadcast required:

1. Enabling `TRACE` on `org.springframework.messaging.simp.broker`,
   `org.springframework.web.socket.messaging`, and
   `org.springframework.messaging.simp.stomp` via the
   `LOGGING_LEVEL_*` environment variable.
2. Reproducing the room-join flow and grepping the log stream for
   `"Broadcasting to N sessions"` produced by Spring's
   `SimpleBrokerMessageHandler`.
3. Reading the subscriber count to confirm that yes, the broker did
   route the frame.

That diagnostic chain was three levels of indirection deep —
re-deploy with new env vars, reproduce, parse Spring's internal
logs — to answer a single yes/no question. The single INFO line
this feature adds at the call site shortens that chain to: grep
the production log for `"Broadcasted RoomJoinedEvent"`. The
TRACE override pattern remains documented as an operator-time
tool for deeper diagnostics (subscriber count, message routing
internals); this feature does not remove it, it complements it.

## How this compares to what I know

- **In Scala / Typelevel this would be...**
  `flatTap(_ => Logger[F].info(s"Broadcasted ... to $destination: ..."))`
  on the success branch of an `IO[Unit]` returned by a
  `messagingTemplate.convertAndSend` wrapper. The
  `EitherT`/`Resource`/IO pattern bakes in the same "log on success
  branch only" structure: the failure branch is its own combinator
  (`handleErrorWith`, `recoverWith`, etc.) and gets the WARN, while
  `flatTap` adds the success observation without touching the value.
  Cats Effect's `Logger[F]` is `INFO`/`WARN`/`ERROR` keyed off the
  same SLF4J levels under the hood, so the operational-vs-developer
  distinction transfers literally.
- **"No shared helper" in Typelevel terms.** A combinator like
  `def broadcastAndLog[A](destination: String, payload: A, ids: ...)`
  in a tagless service trait is the same trap as the
  `broadcastWithLog(Map<String, Object>)` we resisted in Java: it
  collapses heterogeneous shapes into one signature and loses the
  per-event identifiers as compile-time information. The Typelevel
  idiom in that case is to keep the `for { _ <- send; _ <- info }`
  block per call site — explicit, six lines, each with its own
  typed message. Same reasoning, same shape.
- **In Node this would be...** a `winston.info({...})` call placed
  immediately after `socket.emit(...)` resolves. The point about
  operator-readable structured key-value pairs holds verbatim: the
  Node ecosystem just leans harder into JSON-by-default
  (`winston-json`, `pino`), where Spring's default Logback config
  emits plain text. The structured-logging upgrade in either stack
  is a separate concern from "is the log line emitted at the right
  level"; this feature scopes itself to the level question.

## Gotchas / things I learned the hard way

- **Log volume on `MoveEvent` is acceptable but worth naming.** One
  INFO line per move means a hard-played game emits ~80–100 INFO
  lines per game for the broadcast confirmation alone. For a chess
  server with single-digit concurrent games at portfolio scale this
  is well within "operator scrolls through it" territory; for an
  industrial broker pushing millions of events you would push the
  broadcast confirmation to `DEBUG` and rely on metrics
  (Micrometer counters) for the volume signal instead. The decision
  here is right for the scale; documenting it preempts the
  "but won't this be noisy?" review comment.
- **UUIDs in logs are server-generated and not sensitive.** All ids
  emitted by these logs (`gameId`, `playerId`, `joinerId`,
  `winnerId`, etc.) are version-4 UUIDs minted by the server itself,
  uncorrelated with any external identity, and already present
  throughout the rest of the existing log lines (`log.info("Move
  applied: gameId=...")`, `log.info("Room created: roomId=...")`).
  No PII surface change; the policy matches what was already in
  place.
- **The WARN catch is intentionally NOT covered by the new INFO
  log.** The two log lines describe disjoint states: WARN means
  `convertAndSend` threw, INFO means it returned. A failed broadcast
  emits exactly one line (the WARN); a successful broadcast emits
  exactly one line (the INFO). No double-logging on either branch.
  The `try/catch` shape is what guarantees that — the INFO is on the
  line immediately after `convertAndSend` and `convertAndSend`
  cannot return *and* throw.
- **No fully-qualified class names check.** Every log line uses the
  short event-name string (`"RoomJoinedEvent"`, `"MoveEvent"`, etc.)
  as the human-readable label, not the fully-qualified class. The
  events are already imported at the top of each file for the
  `RoomJoinedEvent event = new RoomJoinedEvent(...)` construction;
  no new imports were needed. Grepping the modified files for
  `io\.github\.dariogguillen` outside `import` lines turned up
  nothing new — the convention is respected.
