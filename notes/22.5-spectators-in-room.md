# Feature 22.5 â€” Spectators join the room, not the game (viewer count from the waiting lobby)

**Feature ID:** `spectators-in-room` (from `feature_list.json`)

**Status:** done

## What we built

Re-keyed the live spectator viewer count from the **game** to the
**room**. Previously the count was tracked on `/topic/games/{gameId}`
and published to `/topic/games/{gameId}/viewers` â€” but a `gameId` only
exists once a second player joins, so a creator waiting in the lobby
had nothing for friends to watch. Now the count is tracked on
`/topic/rooms/{roomId}` and published to `/topic/rooms/{roomId}/viewers`,
so spectators can be counted from `WAITING_FOR_PLAYER` onward and the
count stays stable across the `WAITING_FOR_PLAYER â†’ ACTIVE` transition.
The game-keyed `/viewers` topic is retired. No new REST endpoint: the
spectator reuses `GET /api/rooms/{id}` for the initial snapshot and the
room topic for live updates.

## Java / Spring concepts that appear

- **Spring STOMP session events** â€” `SessionSubscribeEvent`,
  `SessionUnsubscribeEvent`, `SessionDisconnectEvent` are application
  events Spring publishes when a STOMP client subscribes, unsubscribes,
  or disconnects. `ViewerCountTracker` listens with `@EventListener`.
  The wrinkle: an `UNSUBSCRIBE` frame carries **only the subscription
  id**, not the destination â€” so the tracker keeps a
  `subscriptionId â†’ (sessionId, roomId)` map populated on subscribe to
  recover which room a later unsubscribe refers to. See
  [Spring WebSocket / STOMP docs](https://docs.spring.io/spring-framework/reference/web/websocket/stomp/application-context-events.html).
- **`StompHeaderAccessor`** â€” wraps a `Message` to read STOMP frame
  metadata (`getDestination()`, `getSubscriptionId()`,
  `getFirstNativeHeader("playerId")`). Native headers are the
  client-supplied STOMP headers, distinct from Spring's own session
  attributes.
- **`SimpMessagingTemplate.convertAndSend`** â€” server-initiated
  broadcast to a topic. The payload (`ViewerCountEvent`, a record) is
  serialized by the configured `MessageConverter` (Jackson) to JSON.
- **Records as immutable map keys / DTOs** â€” `RoomSubscription(sessionId,
  roomId)` is a private record used purely as a value object;
  `ViewerCountEvent(roomId, count)` is the wire DTO. Records give
  structural `equals`/`hashCode` for free, which the
  `stillSubscribed` stream check on unsubscribe relies on.
- **`ConcurrentHashMap` + `newKeySet()`** â€” the three tracking maps are
  concurrent, with set-valued entries backed by
  `ConcurrentHashMap.newKeySet()`. Not a transaction: the
  "is-this-the-last-subscription?" check is racy but converges.

## Decisions taken

- **Decision:** key the count on the room (6-char short code, a
  `String`), not the game (`UUID`).
  - **Alternatives considered:** (a) keep the game keying and add a
    second room-keyed count (dual count); (b) introduce a domain-modelled
    `Spectator` with identity.
  - **Why this one:** the room is the cradle-to-grave lifecycle owner of
    "who is watching this match"; the game is born late. A dual count
    doubles the surface for no product benefit (the frontend has no
    viewer UI yet, so there is exactly one consumer to design for).
    Domain-modelling spectators is overkill for an anonymous,
    fire-and-forget count.

- **Decision:** retire the game-keyed `/topic/games/{gameId}/viewers`
  topic outright rather than keep it alongside the new one.
  - **Alternatives considered:** keep both topics for a deprecation
    window.
  - **Why this one:** this is a **contract retirement without a
    consumer**. No production client subscribes to `/viewers` yet, so the
    breaking change is free. This differs from features 21/22, where old
    frontends were already in production and changes had to be additive.

- **Decision:** `roomId` from the destination is used **verbatim** as
  the map key (no UUID parse), while the `playerId` header is still
  parsed as a `UUID`.
  - **Why this one:** the room id is a short code, not a UUID; the
    player id is a `Player.id()` UUID. A non-UUID `playerId` header is
    treated as "no match" (counted as a viewer); a subscribe to a
    non-existent room is also counted (tolerant â€” mirrors the old
    game-not-found behaviour). The least-surprising failure mode for an
    unauthenticated broadcast layer is "client bug â†’ over-count", never
    "client bug â†’ crash".

- **Decision:** player exclusion via `roomService.findById(roomId)` +
  `room.players()`, matching the trusted `playerId` native header.
  - **Why this one:** `room.players()` is the room-level membership set
    (one player while waiting, two when active), which is exactly the set
    we want to exclude regardless of whether a game exists yet.

## How this compares to what I know

- **In Scala / Typelevel this would be...** the tracking state is a
  `Ref[F, Map[RoomId, Set[SessionId]]]` (plus the inverse `Map[SessionId,
  Set[RoomId]]` and the `Map[SubId, (SessionId, RoomId)]`), updated on a
  stream of session events. Each room's viewers topic is a `Topic[F,
  ViewerCount]` you publish to after every `Ref.update`. The crucial
  parallel is the **consistency model**: this tracker uses a plain
  `ConcurrentHashMap` and a racy "last subscription?" check, which is the
  same "eventually-correct, not transactionally-correct" trade-off you'd
  accept if you used three separate `Ref`s instead of one `Ref` over a
  single product type (or `STM`/`TVar` for a real transaction). The count
  can momentarily be off under tight interleavings but converges on the
  next event â€” acceptable for a spectator badge, unacceptable for, say, a
  balance.
- **In Node this would be...** the `socket.io`/`stomp` server's
  `subscribe`/`unsubscribe`/`disconnect` callbacks mutating a couple of
  `Map`s and emitting on a per-room channel. Same shape, same racy-but-
  fine semantics; Spring just hands you the events as typed
  application-context events instead of raw socket callbacks.

## Gotchas / things I learned the hard way

- The anchored regex (`^/topic/rooms/([^/]+)$`) matters: without the `$`
  anchor, a subscribe to `/topic/rooms/{roomId}/viewers` would itself
  match and self-count. The viewers sub-topic must not bump the count.
- The "stable across join" test asserts the **absence** of a second
  `ViewerCountEvent` after the opponent joins â€” the join touches the
  room state and the game store but never the spectator set, so the
  tracker stays silent and the count is implicitly still 1. Asserting an
  absence (short negative timeout) is the right tool here; there is no
  positive event to wait for.
- Several `@code /topic/games/{gameId}/viewers` references survived in
  unrelated JavaDoc (`RoomEvent`, `GameStateEvent`, `PlayerSessionTracker`)
  and the architecture doc as the canonical "flat single-event topic"
  example. They had to be re-pointed to `/topic/rooms/{roomId}/viewers`
  so the docs don't describe a retired topic.
