# Feature 06.5 — Spectator mode (live viewer count)

**Feature ID:** `spectator-mode` (from `feature_list.json`)

**Status:** in progress

---

## What we built

A `ViewerCountTracker` `@Component` that listens to Spring's STOMP session events
(`SessionSubscribeEvent`, `SessionUnsubscribeEvent`, `SessionDisconnectEvent`) and
maintains a per-game count of spectators — anyone subscribed to
`/topic/games/{gameId}` that did not declare a `playerId` native header matching one
of the game's two players. Every change to the count fires a `ViewerCountEvent` on
a new topic `/topic/games/{gameId}/viewers`. This is the second cross-repo STOMP
contract extension, and it is purely additive: the `MoveEvent` channel from
feature 6 is untouched.

## Java / Spring concepts that appear

- **Spring STOMP session events.** `@EnableWebSocketMessageBroker` causes Spring
  to publish three application events whenever the underlying STOMP session state
  changes: `SessionSubscribeEvent` (a `SUBSCRIBE` frame just arrived),
  `SessionUnsubscribeEvent` (an `UNSUBSCRIBE` frame just arrived), and
  `SessionDisconnectEvent` (the WebSocket transport closed). They are plain
  Spring `ApplicationEvent`s, so any `@EventListener` bean can subscribe to them.
  The catch: `SessionUnsubscribeEvent` does **not** carry the `destination` header
  — only the `subscription-id`. To know which game an unsubscribe affects, the
  tracker has to maintain its own `subscriptionId → (sessionId, gameId)` map
  populated on subscribe. See [Spring's WebSocket events
  reference](https://docs.spring.io/spring-framework/reference/web/websocket/stomp/application-context-events.html).

- **`@EventListener` over `ApplicationListener<T>`.** Spring exposes two ways to
  consume application events: implement `ApplicationListener<SubscribeEvent>` per
  event type, or annotate a regular method with `@EventListener`. The annotation
  form scales better when one bean wants to handle several event types — a
  single `@Component` declares three methods, each with a different event type
  as its sole parameter. The framework wires each to its own listener.

- **`StompHeaderAccessor.wrap(event.getMessage())`** is the canonical way to
  parse STOMP frame metadata server-side. The raw `Message<byte[]>` is a generic
  Spring messaging type; `StompHeaderAccessor` reads the STOMP-specific headers
  out of it — `destination`, `sessionId`, `subscriptionId`, plus native headers
  via `getFirstNativeHeader("playerId")`. "Native" means "headers the client put
  on the frame", as opposed to Spring-internal bookkeeping headers.

- **`ConcurrentHashMap` + `ConcurrentHashMap.newKeySet()` for thread-safe
  set-valued maps.** The tracker maintains three concurrent maps. The
  set-valued ones (`Map<String, Set<String>>`) use
  `computeIfAbsent(k, k -> ConcurrentHashMap.newKeySet())` so that
  initialization of the inner set is atomic with the lookup. Individual map
  operations are thread-safe; multi-step "check then update" sequences (like
  "is this the last subscription of this session to this game?") are racy in
  the strict-consistency sense but converge for our broadcast semantics.

- **Trust-on-self-declaration identity.** The `playerId` STOMP header is a
  client claim the server does not verify. Today the server has no auth layer,
  so verification is impossible. Tomorrow, when auth lands, the validation
  changes (compare claim to authenticated principal) but the wire shape and the
  exclusion logic do not. This is a deliberate portfolio trade-off, documented
  in `docs/architecture.md`.

## Decisions taken

**STOMP `playerId` native header for self-identification, not a REST handshake.**

- Decision: clients that are players send `playerId:<uuid>` as a native STOMP
  header on the `SUBSCRIBE` frame to `/topic/games/{gameId}`. The server reads
  it via `StompHeaderAccessor.getFirstNativeHeader` and compares against
  `white.id()` / `black.id()` of the game.
- Alternatives considered: (a) a `Viewer` domain record + REST endpoint to
  register as a viewer; (b) infer "is this a player" from a session-to-player
  map populated by a separate `CONNECT` flow; (c) ignore the player/spectator
  distinction and just report the total subscriber count, with the frontend
  doing "minus the players I happen to know about".
- Why: zero new REST surface, no new domain models, no server-side session
  bookkeeping beyond what the event listeners already track. The server's
  exclusion logic is one comparison. When auth is added the header becomes a
  verified principal instead of a claim — same wire, smarter validation.

**Separate `/viewers` sub-topic, not a `viewerCount` field embedded in
`MoveEvent`.**

- Decision: a brand-new topic `/topic/games/{gameId}/viewers` carrying a small
  `ViewerCountEvent(gameId, count)`.
- Alternatives: extend `MoveEvent` with a `viewerCount` field; emit count
  updates on the same `/topic/games/{gameId}` channel with a discriminator
  field.
- Why: viewer-count changes are not tied to move cadence — a game can have
  zero moves and lots of join/leave activity. Embedding the count in
  `MoveEvent` would couple the two. A separate topic also lets clients
  subscribe only to count changes (cheap UI badge update) without paying for
  the move stream, and the message-broker fan-out works the same way.

**`subscriptionId → (sessionId, gameId)` map for unsubscribe handling.**

- Decision: maintain a third map keyed by subscription id, holding a
  `GameSubscription(sessionId, gameId)` record. Populated on subscribe,
  consulted on unsubscribe, swept on disconnect.
- Alternatives: only listen to `SessionDisconnectEvent` and rely on it for
  cleanup; reconstruct the destination from the session-and-subscription via
  the broker's internal subscription registry.
- Why: `SessionUnsubscribeEvent` does not carry `destination`, only
  `subscription-id`. Without the map, the tracker cannot decrement the count
  for a "stays connected, unsubscribes from the game topic" path — which is
  realistic in a SPA where the user navigates from a game page to a lobby
  without closing the WebSocket. Relying solely on disconnect would miss this
  case. Reaching into the broker's internal registry is a coupling we do not
  want.

**Regex anchored at `$` so `/viewers` itself is not a game topic.**

- Decision: `^/topic/games/([^/]+)$` — the trailing `$` is intentional.
- Alternatives: `^/topic/games/([^/]+)` (no end anchor) which would match
  `/topic/games/{id}/viewers` and count viewer-of-viewers as a viewer.
- Why: subscribers to `/topic/games/{id}/viewers` are watching the count, not
  watching the game. Counting them would create a feedback loop (every new
  viewer-of-viewers triggers a count change, which delivers to all the
  viewers-of-viewers, who do not affect the count). End-anchoring keeps the
  game topic and its `/viewers` sub-topic as cleanly separated namespaces.

**Same-session deduplication via set semantics on `sessionsByGame`.**

- Decision: `sessionsByGame` is keyed by `sessionId` (not `subscriptionId`),
  with `ConcurrentHashMap.newKeySet()` values. Multiple subscriptions from the
  same session to the same game count as one viewer.
- Alternatives: count subscriptions, so a misbehaving client that sends two
  SUBSCRIBEs inflates the count by two.
- Why: a viewer is a person/browser, not a frame. Set semantics make the
  count match user intuition.

**Test helper duplication, not extraction.**

- Decision: `setupGame(...)` and `baseUrl()` are copy-pasted from
  `GameWebSocketIT` into `ViewerCountIT`.
- Alternatives: extract a shared `GameSetupHelper` under
  `src/test/java/.../testsupport/`.
- Why: two callers is the cutoff where extracting starts to pay off, and the
  helpers are small (~30 lines together). The two ITs also have different
  STOMP needs and may diverge in setup (e.g. one of them adding a custom
  STOMP header before connect); a premature extraction would freeze a shape
  that doesn't fit yet. We accept the duplication and revisit if a third
  caller appears or the helpers stabilize for two more features.

## How this compares to what I know

- **vs Akka cluster pub/sub presence tracking.** In Akka, you would model
  this with a `DistributedPubSub` mediator and `DistributedPubSubMediator.GetTopics`
  / topic subscriber counting via the cluster receptionist. Spring's
  `SimpleBroker` is single-node, so the tracker maintains the count in-process
  with `ConcurrentHashMap`. The conceptual move — "watch the
  subscribe/unsubscribe stream, derive a presence set" — is the same; the
  scaling cliff in Spring is "one JVM", in Akka it is "the cluster boundary".

- **vs fs2 `Topic[F, A].subscribers` / `cats.effect.std.MapRef`.** `fs2.Topic`
  exposes a subscriber count via `subscribers: Stream[F, Int]`. The shape is
  the same: pub/sub primitive + a derived count stream. The difference is
  that with fs2 you have one process and one `Topic` value you pass around;
  in Spring you have a broker bean and an `@EventListener` that hooks the
  pub/sub layer's lifecycle events. Both end at the same place — a
  monotonically updated integer broadcast to interested parties.

- **vs Node `socket.io` `io.in(room).fetchSockets()`.** socket.io's
  per-room socket adapter exposes the connected set; the count is
  `(await io.in(room).fetchSockets()).length`. The equivalent in Spring is
  what this tracker computes. Both worlds expose the subscribe/unsubscribe
  events; both make the count a derived value, not a stored one. The Node
  flavor is more imperative ("ask for the set now"); the Spring flavor is
  reactive ("update on every event"). Either could be implemented the other
  way around.

## Gotchas / things I learned the hard way

- **`SessionUnsubscribeEvent.getMessage()` does not carry the `destination`
  header.** The first attempt at the tracker tried to read `destination` on
  unsubscribe to find the game id, like on subscribe. It is always null. STOMP
  `UNSUBSCRIBE` frames carry only the `subscription-id`, by spec. The
  workaround is the `subscriptionToGame` map populated on subscribe.

- **Multi-subscription dedupe is per session-and-game pair, not global.** The
  tracker's "is this the last subscription of this session to this game?"
  check looks at `subscriptionToGame.values()` filtering by both `sessionId`
  and `gameId`. The first cut filtered by `sessionId` alone and decremented
  too eagerly when a session had two subscriptions to *different* games and
  unsubscribed from one.

- **The regex must be anchored at `$`.** Without the trailing `$`,
  `/topic/games/{id}/viewers` matches the pattern and the viewer-of-viewers
  subscription bumps the count. The first IT run showed counts of 2 instead
  of 1 because every subscribe to `/viewers` was counted on top of the
  subscribe to `/topic/games/{id}`. The fix is one character.

- **STOMP `SUBSCRIBE` headers vs `CONNECT` headers are different surfaces.**
  Spring's `WebSocketStompClient` lets you pass `WebSocketHttpHeaders` on
  `connectAsync(url, headers, handler)`, but those become CONNECT-frame
  headers, not SUBSCRIBE-frame headers. The tracker reads `playerId` from the
  SUBSCRIBE frame's native headers, so the IT uses the
  `session.subscribe(StompHeaders, handler)` overload and calls
  `headers.add("playerId", value)`.

- **Order of events matters in tests.** The IT subscribes to `/viewers`
  *before* subscribing to the game topic — otherwise the count broadcast that
  the subscribe triggers misses the viewer-side subscription, the queue stays
  empty, and the test times out for the wrong reason.
