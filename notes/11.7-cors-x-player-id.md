## Feature 11.7 — Allow X-Player-Id header in REST CORS

**Feature ID:** `cors-x-player-id` (from `feature_list.json`)

**Status:** in progress

---

## What we built

A one-line bug fix in `CorsConfig`: the `allowedHeaders` list now
includes `X-Player-Id` next to `Content-Type` and `Accept`, plus a
new `CorsConfigIT` test that locks the entry in. With this change
the browser preflight for `POST /api/games/{id}/moves` from a
cross-origin frontend stops being blocked, and the production E2E
flow (create room → join → move) actually completes from
`https://dariogguillen.github.io`.

## Java / Spring concepts that appear

- **`@RequestHeader` requirements as an implicit CORS contract.**
  Spring's `@RequestHeader("X-Player-Id")` says nothing about CORS
  at the framework level — it just binds the header value into a
  controller parameter. But cross-origin, any custom header
  (anything not in the small CORS-spec "safe-listed" set:
  `Accept`, `Accept-Language`, `Content-Language`, and a narrow
  subset of `Content-Type`) **forces** the browser to send a
  preflight OPTIONS that asks "may I include this header?". If
  the server's `Access-Control-Allow-Headers` doesn't echo the
  header back, the browser rejects the entire cross-origin call
  before the real request goes out. So a `@RequestHeader` on a
  custom name like `X-Player-Id` is an implicit requirement on
  the CORS allow-list that the type system does not surface. See
  [MDN — CORS preflighted requests](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#preflighted_requests).
- **Browser "simple headers" vs "preflighted headers".** The
  fetch spec calls a request "simple" (no preflight) when the
  method is `GET`/`HEAD`/`POST`, the headers are all CORS-safe-
  listed, and the `Content-Type` (if present) is one of three
  narrow values. The moment you add a custom header, the request
  is preflighted. This is the asymmetry that hides the bug from
  same-origin tests: a `MockMvc` integration test runs same-origin
  by construction, the CORS filter is invoked, but the **real
  request** in `MockMvc` does not need a preflight to pass — only
  the actual browser does. See
  [Fetch spec — CORS-safelisted request-header](https://fetch.spec.whatwg.org/#cors-safelisted-request-header).
- **Why E2E catches what unit / IT misses.** The existing
  `CorsConfigIT` preflight tests requested `Content-Type` only —
  the same list the allow-list configured. They were a
  tautology: they tested that the value present in
  `allowedHeaders` was echoed back. They could never catch a
  header *missing* from the allow-list, because the test request
  did not ask for that header. Only a real cross-origin browser
  request, against the actual endpoint that requires
  `X-Player-Id`, exercises the gap. The new test in this feature
  ports that browser intuition into the IT layer by explicitly
  preflighting against `/api/games/{gameId}/moves` with
  `Content-Type, X-Player-Id`.

## Decisions taken

**Add `X-Player-Id` to the existing list, not a `"*"` wildcard.**

- Decision: extend the explicit allow-list to
  `Content-Type, Accept, X-Player-Id`.
- Alternatives: `.allowedHeaders("*")` to allow everything.
- Why: wildcard `"*"` would silently widen the surface every
  time a new custom header lands anywhere in the codebase. The
  point of an allow-list is that it forces a deliberate edit
  paired with the feature that introduces the requirement. Same
  rationale that feature 10 used to keep `Authorization` off:
  the list is documentation, not just a security knob. One
  explicit string per known header is cheap; a wildcard erases
  the trail.

**Keep `Authorization` deliberately out.**

- Decision: do not add `Authorization` even though we are
  touching this list.
- Alternatives: throw it in "for forward compatibility" now that
  we are editing the line anyway.
- Why: the rationale from feature 10 still holds — there is no
  auth header anywhere in the codebase, allow-listing it would
  imply functionality we have not built, and the future auth
  feature owns adding it as part of its own change. Resisting
  the temptation to piggy-back is the discipline.

**Test against `/api/games/{gameId}/moves` specifically.**

- Decision: the new preflight test targets the moves endpoint
  with a random UUID in the path.
- Alternatives: add the test against `/api/rooms` (the existing
  preflight target) — CORS is path-pattern-based via `/api/**`,
  so any path under that prefix would hit the same filter and
  produce the same response headers.
- Why: the bug is endpoint-specific in spirit even though the
  CORS mapping is broad. Documenting the test against the
  affected endpoint makes the regression intent visible: a
  reader scanning `CorsConfigIT` sees `/api/games/.../moves` and
  immediately understands which production flow this guards. The
  test would functionally pass against `/api/rooms` too, but the
  signal it carries to a future reader would be weaker. Tests
  are documentation; the path-name is part of the documentation.

## How this compares to what I know

- **In Scala / Typelevel this would be...** the same shape of
  bug that you'd hit if you added a new field to a request type
  but forgot to extend its `Codec` / `Encoder` (or, more
  symmetrically, added a new route variant to a sealed
  hierarchy but forgot to register it in the route table). The
  difference is the catch-time. In tagless-final / whole-shape
  derived code, the compiler does the cross-check for you —
  there is no path between "the route reads field X" and "the
  codec decodes field X" that isn't anchored on the same
  derivation, so dropping one end fails to compile. Spring's
  annotation-driven model decouples `@RequestHeader("X-Player-Id")`
  from `.allowedHeaders(...)` entirely: they live in different
  classes, with no shared type, and the framework has no way of
  knowing they describe two ends of the same contract. The
  trade-off is the usual one: the annotations buy quick
  prototyping at the cost of contracts that are only checked at
  runtime (or, as here, only checked in the browser).
- **In Node this would be...** the analogous bug with
  `cors({ allowedHeaders: ['Content-Type', 'Accept'] })` plus an
  Express handler reading `req.headers['x-player-id']`. The two
  lines are equally decoupled; the catch is the same browser
  preflight. The Node ecosystem typically discovers this with
  Cypress / Playwright, the way our `chess-frontend` discovered
  it for us here.

## Gotchas / things I learned the hard way

- **The cross-feature gap was invisible inside each feature.**
  Feature 5 added `@RequestHeader("X-Player-Id")` on the moves
  endpoint. Feature 10 declared the CORS allow-list. Both
  features were green in isolation, both had passing ITs, and
  the gap only surfaced when the frontend ran in a real browser
  against a real cross-origin URL. The lesson is structural:
  whenever a feature adds a custom request header on a REST
  endpoint, the CORS allow-list is part of that feature's
  surface, even if CORS is owned by a different config class.
- **Same-origin MockMvc cannot reproduce a browser preflight
  failure naturally.** `MockMvc` is happy to send any header you
  hand it; the CORS filter runs but it never blocks the real
  request the way a browser would. The new test simulates the
  preflight explicitly with an `OPTIONS` + `Access-Control-
  Request-Headers`; without doing that step by hand, the test
  framework will never tell you the allow-list is wrong.
- **The existing IT was a tautology.** The pre-fix preflight
  tests asked for `Content-Type` and asserted `Content-Type` was
  echoed back. They could not have failed if the allow-list was
  `Content-Type` only — they could only fail if the allow-list
  dropped `Content-Type` outright. A test that locks in
  *exactly* what the production code says, without exercising
  the boundary, is a test that protects you from one specific
  regression and lulls you on every other.

  and which headers force it.
