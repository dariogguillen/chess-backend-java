# Feature 23.5 — Choose the bot's strength by Elo (Stockfish UCI_LimitStrength)

**Feature ID:** `bot-difficulty` (from `feature_list.json`)

**Status:** done

---

## What we built

Feature 23 shipped a bot that always played at one fixed strength. This
feature lets the room creator pick the bot's strength as a **target Elo**
when creating a `opponentKind=BOT` room. The chosen Elo rides on the
`Game` (a new nullable `Integer botElo`, threaded `CreateRoomRequest →
Game` exactly like `timeControl`) and is applied on every bot move via
Stockfish's `UCI_LimitStrength` + `UCI_Elo` engine options. Omitting the
field falls back to a configured default (`chess.bot.default-elo`); an
out-of-range value is a 400 `VALIDATION_FAILED`.

## Java / Spring concepts that appear

- **UCI engine options (`setoption`)** — UCI (Universal Chess Interface)
  is the line-protocol Stockfish speaks over stdin/stdout. Options are set
  with `setoption name <X> value <Y>` on a *ready* engine (after
  `isready`) and *before* the search (`go`). We send
  `UCI_LimitStrength=true` (flip the engine from "strongest move" into
  Elo-limited mode) and `UCI_Elo=<elo>` (the target). The engine **clamps
  internally** to its own build's range, so any in-range value is safe to
  send. See `StockfishBotEngine.chooseMove`. UCI spec:
  <https://backscattering.de/chess/uci/>.

- **Bean Validation for a bounded numeric input (`@Min`/`@Max`)** — on the
  DTO component `CreateRoomRequest.botElo`, `@Min(BotEngine.MIN_BOT_ELO)`
  + `@Max(BotEngine.MAX_BOT_ELO)` declare the legal range. With
  `@Valid @RequestBody` on the controller, an out-of-range value is
  rejected at the boundary and `GlobalExceptionHandler` maps the
  `MethodArgumentNotValidException` to a 400 `VALIDATION_FAILED` — no new
  error code, no service-layer `if`. Annotation attributes must be
  *compile-time constants*, which is why `MIN_BOT_ELO`/`MAX_BOT_ELO` are
  `static final int` on `BotEngine` (a constant expression is legal in an
  annotation; a method call would not be). The **same two constants**
  validate `BotProperties.defaultElo` in its compact constructor, so the
  request bound and the configured default can never drift apart. Jakarta
  Bean Validation:
  <https://jakarta.ee/specifications/bean-validation/3.0/>.

- **Backwards-compatible record evolution** — `Game` is an immutable
  `record`. Adding `botElo` as the **last** canonical component, plus a
  convenience constructor that delegates with `null`, keeps every existing
  `new Game(...)` call site compiling and lets Jackson deserialise
  pre-deploy Redis games (which lack the field) as `botElo = null`.
  `withStatus` / `withClock` carry it through. This is the exact trick
  feature 22 used for the clock fields and feature 16 for `Player.userId`.

- **Widening a port (interface) + its adapters in lockstep** — `BotEngine`
  is a hexagonal *port* with one production adapter
  (`StockfishBotEngine`), one Mockito test double, and a gated real-binary
  IT. Changing `chooseMove(String)` to `chooseMove(String, int)` is a
  single-seam change: update the interface, the adapter, and every
  Mockito `when(...)` / `verify(...)` site together. The compiler finds
  them all because there is exactly one method.

- **`@ConfigurationProperties` record with compact-constructor validation**
  — `BotProperties` binds `chess.bot.*` and validates `defaultElo ∈
  [MIN, MAX]` at *bean-binding* time, so a misconfigured default fails the
  boot fast rather than surfacing as a mid-game surprise.

## Decisions taken

- **Decision:** Elo (a single integer) instead of three named tiers
  (Easy/Medium/Hard).
  **Alternatives considered:** discrete tiers mapped to Elo values.
  **Why this one:** the user's call — three tiers jump too coarsely, and
  `UCI_Elo` is already a continuous knob. A slider/select in the frontend
  maps 1:1 to the field.

- **Decision:** the Elo lives on the `Game`, not just on the `Room`.
  **Alternatives considered:** store it only on the `Room` and look it up
  per move.
  **Why this one:** `BotMoveService` reads the authoritative `Game` on
  every move and must apply the right strength each time; putting the Elo
  on the `Game` avoids a cross-aggregate `Room` lookup on the hot path,
  exactly as feature 22 lifted `incrementMs` onto the `Game`.

- **Decision:** bounds live on the DTO (`@Min`/`@Max`); the null→default
  fallback lives in the service.
  **Alternatives considered:** a service-layer range check; defaulting in
  the DTO.
  **Why this one:** the *bound* is a contract fact the OpenAPI spec should
  advertise and the framework should enforce uniformly (400). The
  *default* is a deployment concern (`chess.bot.default-elo`), and a DTO
  cannot read configuration — so resolving `null → default` belongs in
  `RoomService` (and, defensively, in `BotMoveService` for an in-flight
  game whose `botElo` is null).

- **Decision:** no Postgres persistence of the Elo (no Flyway migration).
  **Why this one:** the active Redis `Game` is enough to play at the right
  strength; "vs Stockfish (1500)" in game *history* is a separable
  follow-up that would touch `GameEntity` + a migration. The
  `GameEntityMapper` simply does not map `botElo`.

- **Decision:** the ~1320 floor is documented, not worked around.
  **Why this one:** below ~1320, Stockfish strength is governed by `Skill
  Level`, not `UCI_Elo`. Supporting sub-1320 beginner play is a distinct
  follow-up; bounding the field at 1320 keeps the model honest.

## How this compares to what I know

- **In Scala / Typelevel this would be...** the bounded Elo is a textbook
  **refined type**: `Refined[Int, Interval.Closed[1320, 3190]]` (or an
  `iron` constraint), validated *at the edge* when decoding the request
  body, so the rest of the program carries a value that is *correct by
  construction* — no `@Min`/`@Max` runtime check threading through.
  `@Min`/`@Max` is the imperative analogue: the validation runs
  reflectively after Jackson binds the record, and a violation throws.
  Widening `chooseMove(fen)` to `chooseMove(fen, elo)` is the same as
  adding a parameter to a method on a **tagless-final algebra**
  (`trait BotEngine[F[_]] { def chooseMove(fen: Fen, elo: Elo): F[Move] }`)
  — every interpreter (the real one, the test stub) updates in lockstep,
  and the compiler enforces it. The `BotProperties` record is a
  `Ciris`/`pureconfig`-decoded config case class whose smart constructor
  rejects an out-of-range default.

- **In Node this would be...** the bound would be a Zod/Joi schema on the
  request body (`z.number().int().min(1320).max(3190).optional()`),
  rejecting out-of-range input with a 400 in the validation middleware —
  the direct analogue of `@Min`/`@Max` + the global exception handler. The
  Stockfish subprocess would be driven with `child_process.spawn` over the
  same UCI line protocol; the `setoption` lines are byte-identical
  regardless of host language.

## Gotchas / things I learned the hard way

- `@Schema(minimum = ...)` and the `@Min`/`@Max` *values* must be
  compile-time constants. `"" + BotEngine.MIN_BOT_ELO` is a constant
  String expression (constant + constant), so it is legal in the
  annotation; anything involving a method call would not compile.
- Spotless line-wraps JavaDoc and comments aggressively — `mvn
  spotless:apply` is part of the loop, not an afterthought; a "compile
  error" was really a formatting violation the first time around.
- The test profile inherits `chess.bot.default-elo: 1500` from
  `application.yml` (no override in `application-test.yml`), so the
  `null botElo → 1500` IT assertion is pinned to that value.
