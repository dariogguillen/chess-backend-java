# Feature 23.94 — Single archived game with its full move list (replay)

**Feature ID:** `game-review` (from `feature_list.json`)

**Status:** done

---

## What we built

A new authenticated read endpoint, `GET /api/me/games/{id}`, that
returns ONE archived game with its FULL ordered move list (plus the
starting and final FEN) so the frontend can replay the game move by
move. The existing `/api/me/games` list and `/api/me/stats` aggregate
only ever expose a `moveCount`; this is the first surface that ships the
moves themselves. The caller must be a participant of the game; a
non-participant, an unknown id, and an anonymous game all return the same
`404 GAME_NOT_FOUND` (no leak). This is the fourth and last feature of
the profile-support arc (`profile-edit → game-result-persistence →
me-stats → game-review`) and **completes the arc**.

## Java / Spring concepts that appear

- **`LEFT JOIN FETCH` on a lazy association** — The `moves` collection
  on `GameEntity` is `@OneToMany(fetch = FetchType.LAZY)`. With
  `spring.jpa.open-in-view = false` (set project-wide), the Hibernate
  session is closed by the time a controller maps the entity, so a lazy
  collection accessed there throws `LazyInitializationException`. The
  fix is to fetch the collection inside the read transaction with a
  fetch-join: `SELECT g FROM GameEntity g LEFT JOIN FETCH g.moves WHERE
  …`. `JOIN FETCH` (as opposed to a plain `JOIN`) tells Hibernate to
  initialise the association in the same SQL round-trip, not to issue a
  second query.
  [Hibernate fetch strategies](https://docs.jboss.org/hibernate/orm/6.4/userguide/html_single/Hibernate_User_Guide.html#fetching).
- **`LEFT` vs inner `JOIN FETCH`** — An inner `JOIN FETCH` only returns
  parent rows that have at least one child row. A game that terminated
  before any move (a timeout/abandon with zero moves) has no rows in
  `moves`, so an inner join would silently drop it — the participant
  would get a spurious 404. `LEFT JOIN FETCH` keeps the parent and
  yields an empty collection. A dedicated IT seeds a zero-move game and
  asserts `200` + `moves: []`, so a regression to `INNER` fails the
  suite.
- **`@AuthenticationPrincipal` + `@PathVariable`** — The handler takes
  `@AuthenticationPrincipal User currentUser` (the JWT subject resolved
  by the security filter chain; non-null by construction because the
  `/api/me/**` chain 401s anonymous calls before the controller runs)
  and `@PathVariable("id") UUID id`. Spring binds the path segment
  straight to a `UUID`.
- **`@Transactional(readOnly = true)` spanning the entity→DTO mapping** —
  The mapping from `GameEntity` to `MyGameDetail` runs *inside* the
  service method, not in the controller, precisely so the persistence
  context is still open when the (already-fetched) `moves` are read.
  `readOnly = true` is a hint that lets Hibernate skip dirty-checking /
  flush.
- **`@OrderBy("moveIdx ASC")`** — declared on the association, so the
  fetched moves arrive in playback order and the service maps them with
  a plain `stream().map(...)`, no re-sort.

## Decisions taken

- **Decision:** Reuse `GameStateResponse.MoveSummary`
  (`web.game`) as the per-move element of `MyGameDetail` (`web.me`)
  rather than defining a fresh move record in `web.me`.
  - **Alternatives considered:** a local `web.me`-private move record,
    to keep the two packages decoupled.
  - **Why this one:** the wire shape is byte-for-byte identical
    (`{ from, to, promotion }`, lowercase algebraic squares, nullable
    `Piece` enum name). A second record would be a copy with no semantic
    difference, and it would force the frontend to carry two parsers for
    one concept. One DTO keeps the live-game and archived-game replay
    surfaces uniform. The cross-package import is acceptable here — the
    move shape is a shared wire vocabulary, not an internal of either
    controller.

- **Decision:** Fold the authorization into the query's `WHERE` clause
  (`g.id = :gameId AND (g.whiteUserId = :userId OR g.blackUserId =
  :userId)`) and map the empty result to `404 GAME_NOT_FOUND`.
  - **Alternatives considered:** load the game by id, then check
    participation in the service and throw `403` for a non-participant.
  - **Why this one:** a `403` confirms the game exists, letting an
    attacker probe for valid game ids. Collapsing "doesn't exist", "not
    a participant", and "anonymous game" into one indistinguishable
    `404` is the no-leak contract — the exact same discipline the
    friends endpoints use (`FRIEND_REQUEST_NOT_FOUND`). It also reuses
    the existing `GameNotFoundException` / `GAME_NOT_FOUND` code, so the
    `ErrorResponse.allowableValues` canary stays at 21 (no new code).

- **Decision:** No Flyway migration.
  - **Why:** every column the endpoint reads (`white_user_id`,
    `black_user_id`, `result`, `final_fen`, the `moves` table) already
    exists from earlier features. This is a pure read surface; V4 stays
    the latest migration.

## How this compares to what I know

- **In Scala / Typelevel this would be...** The `LEFT JOIN FETCH` is
  the declarative-ORM stand-in for what you write by hand in doobie:
  there, you'd express the join explicitly in the SQL
  (`LEFT JOIN moves m ON m.game_id = g.id`) and assemble the
  `Game(..., moves)` aggregate from the result set yourself — the
  `LEFT` vs inner choice is right there in the SQL, fully visible. JPA
  hides the join behind a lazy association, and the trap is that the
  laziness + `open-in-view = false` means "forgot the fetch-join" fails
  at *read time* with a runtime exception, not at compile time. doobie's
  model is more honest (the query is the contract); JPA's is terser
  until it bites. The fix — being explicit about the fetch — converges
  back to the doobie shape.
- **Authorization-in-the-WHERE-clause vs a separate guard.** In an
  http4s app you'd typically thread the authenticated user through and
  write a `Repository.findGame(id).flatMap(g => if (g.belongsTo(user))
  ... else NotFound)` — a guard *after* the load. Baking the predicate
  into the query is the same idea pushed down a layer: the database
  never returns a row the caller isn't entitled to, so there is no
  in-between state where "I have the game but you may not see it" can
  leak through a wrong status code. It's the SQL equivalent of an
  `EitherT`-style short-circuit where the "not authorized" and "not
  found" branches are deliberately merged.
- **In Node this would be...** an Express handler doing
  `repo.findOne({ id, $or: [{ whiteUserId }, { blackUserId }] })` with a
  populated `moves`, returning 404 on null — structurally the same
  participant-scoped query. The Java version's value-add is that the
  lazy-vs-eager fetch is a typed, testable seam (the zero-move IT
  pins it).

## Gotchas / things I learned the hard way

- The `LEFT`-not-inner distinction is invisible until you have a
  zero-child row. Without the dedicated zero-move IT, an inner
  `JOIN FETCH` would pass every other test and only fail in production
  on the first timeout-before-first-move game. The test is the spec.
- `spring.jpa.open-in-view = false` is the project's standing setting,
  and it is the whole reason the fetch-join is mandatory rather than
  optional — with OSIV on (the Spring Boot default), the lazy access in
  the controller would have *worked*, masking the issue. Knowing the
  flag's value is load-bearing for getting the query right.
