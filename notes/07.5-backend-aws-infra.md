# Feature 07.5 — Provision AWS infrastructure for the backend

**Feature ID:** `backend-aws-infra` (from `feature_list.json`)

**Status:** in progress

---

## What we built

A declarative AWS Free Tier deployment for the backend: Terraform in
`infra/` provisions an EC2 t3.micro on Ubuntu 24.04 LTS, an RDS Postgres
db.t3.micro, an ECR repository, an Elastic IP, two Security Groups, and
a $1/month Budget alarm. Cloud-init installs Docker and Caddy on first
boot, creates a `deploy` user, and writes the Caddyfile from a Terraform
template. A new `docker-compose.prod.yml` at the repo root runs Redis +
the app container against the RDS endpoint and binds the app to loopback
only so Caddy is the sole inbound path. The first deploy is manual,
documented step by step in `docs/deploy-runbook.md`; feature 7.7 will
automate it via GitHub Actions.

## Java / Spring concepts that appear

This feature is infrastructure, not Java code, so the "concepts" are
DevOps idioms rather than `@Annotation` machinery. They still matter:
they shape the contract the Spring Boot image needs to honor at
runtime.

- **Terraform providers, resources, variables, outputs, state.**
  Terraform's model is a small, well-named handful of primitives.
  A `provider` block configures a backend (AWS, GCP, etc.). A
  `resource` declares a desired-state object (an EC2 instance, an
  RDS instance). A `variable` is the input knob, with a type and
  optional default. An `output` is the value Terraform prints after
  apply, conventionally used as the hand-off to whatever runs next
  (a CI step, a runbook reader). The `state` file is what
  Terraform keeps locally (or remotely, in S3 + DynamoDB lock — not
  done here) that maps "the names I declared" to "the cloud objects
  I created". `terraform plan` diffs declared vs state-known
  resources; `terraform apply` makes the API calls to close the
  gap. See the [Terraform language overview](https://developer.hashicorp.com/terraform/language).

- **AWS resource model: VPC, subnets, SGs, EIP, EC2, RDS.** Every
  AWS account ships with a **default VPC** per region — a pre-built
  network with one public subnet per Availability Zone. Using it
  skips the custom-VPC ceremony (subnet design, route tables, NAT
  gateways) that has its own learning curve. A **Security Group**
  is a stateful firewall attached to one or more resources; rules
  list source CIDRs (`0.0.0.0/0` = anywhere, `1.2.3.4/32` = one IP)
  or other SGs as sources (`rds` ingress from `ec2` SG = "only
  hosts in the EC2 SG can hit Postgres"). An **Elastic IP** is a
  static public IPv4 you allocate to your account and attach to an
  instance; AWS charges for unattached EIPs (a Free Tier pitfall).
  An **EC2 instance** is a VM; `t3.micro` is the Free Tier baseline.
  **RDS** is managed Postgres; `db.t3.micro` + 20 GB GP3 storage +
  single-AZ is the Free Tier config. See
  [AWS default VPC docs](https://docs.aws.amazon.com/vpc/latest/userguide/default-vpc.html)
  and the [Free Tier landing
  page](https://aws.amazon.com/free/).

- **`cloud-init` as the standard "first boot" hook.** Linux cloud
  images (Ubuntu, Amazon Linux, Debian, etc.) ship with cloud-init
  pre-installed. AWS exposes the `user_data` field on `aws_instance`;
  on first boot, cloud-init reads `user_data` and executes it. The
  contract is "this script runs once, as root, after the network is
  up and a few base services have started, before the SSH banner
  unblocks". Terraform's `templatefile()` function renders the
  script at plan time so values like the Duck DNS subdomain and the
  deploy user's public key are interpolated before the bytes ever
  hit AWS. The phases (`init`, `config`, `final`) and the log
  locations (`/var/log/cloud-init-output.log` for the merged stdout,
  `/var/log/cloud-init.log` for the framework log) are the two
  things you need to know to debug a failed first boot. See
  [cloud-init documentation](https://cloudinit.readthedocs.io/).

- **Caddy and the HTTP-01 ACME challenge.** Caddy obtains TLS
  certificates from Let's Encrypt automatically the first time a
  hostname is requested. The default challenge type is HTTP-01:
  Let's Encrypt opens port 80 against the hostname's A record and
  serves a token from a well-known path; Caddy proves it controls
  the hostname by responding correctly. This is **why the EC2
  Security Group must allow inbound port 80 even though the app
  speaks HTTPS only** — without port 80 reachable from the public
  internet, Let's Encrypt cannot validate the hostname and the
  certificate never issues. Caddy then automatically redirects
  HTTP → HTTPS once the cert is in place and refreshes it ~30 days
  before expiry. See [Caddy's automatic HTTPS
  docs](https://caddyserver.com/docs/automatic-https) and the
  [Let's Encrypt HTTP-01 challenge
  reference](https://letsencrypt.org/docs/challenge-types/#http-01-challenge).

- **The env-var-with-default pattern from feature 7, paying off.**
  `application.yml` reads `SPRING_DATASOURCE_URL` and
  `SPRING_DATA_REDIS_HOST` from the environment with sensible
  localhost defaults. In production, `/opt/chess/.env` overrides
  both — `SPRING_DATASOURCE_URL` points at the RDS endpoint,
  `SPRING_DATA_REDIS_HOST` points at the Redis container in the
  Docker network. **Zero code or image changes** between local and
  production. The Spring Boot image built locally is the exact
  image that runs in EC2. This is the textbook payoff of
  twelve-factor config separation; it took one feature to set up
  and the next feature is where you collect on it.

- **Manual deploy before automation, as a learning principle.**
  Feature 7.7 will automate every step from "build image" through
  "smoke test the live endpoint". Automating it before doing it by
  hand once is a recipe for opaque CI failures — the first time
  something goes wrong, you have to debug both the deploy and the
  automation simultaneously. Doing the runbook manually pins the
  contract: every step has a known-good output, every failure mode
  has a known-good recovery, and the CI workflow is then a
  mechanical translation of a process the developer has run end to
  end. The runbook in `docs/deploy-runbook.md` is the artifact that
  makes the next feature's automation tractable.

## Decisions taken

**Terraform over the AWS Console (or a shell script).**

- Decision: provision all AWS resources via Terraform, with a
  declarative description checked into the repo.
- Alternatives: click through the AWS Console (fast, opaque,
  unreproducible); a Bash script that calls the AWS CLI (imperative,
  hard to keep idempotent, no drift detection); CloudFormation
  (AWS-native, more verbose, AWS-only); CDK / Pulumi (programming-
  language IaC; great but a steeper learning curve).
- Why: Terraform is the lingua franca of cloud IaC in 2026 — the
  CV signal premium alone justifies the few hours of learning. The
  declarative model also matches the "destroy + apply regenerates
  everything" property we want for cycling through AWS Free Tier
  expiry events: a year from now, we re-`apply` against a different
  AWS account and the stack reproduces.

**Caddy over Nginx + Certbot.**

- Decision: Caddy, native systemd, with a 3-line Caddyfile.
- Alternatives: Nginx + Certbot (the historical default); Traefik
  (great for dynamic Docker discovery but heavier); HAProxy (great
  for L4 + sticky sessions but no built-in ACME).
- Why: Caddy bundles ACME + reverse proxy + HTTP/2 + HTTP/3 +
  automatic HTTPS into a single binary with a configuration that
  fits on one screen. The Nginx + Certbot path is three configs
  (the Nginx site, the Certbot cron, the renewal hook), each
  with its own gotcha. The Caddyfile in this feature is six lines.
  The pattern transfers — anyone who can read this Caddyfile can
  read an Nginx config — but the operational simplicity is hard to
  match.

**Default VPC.**

- Decision: use the account's default VPC + default subnets.
- Alternatives: create a custom VPC with explicit public + private
  subnets, NAT gateway, IGW, etc.
- Why: a custom VPC is a learning exercise on its own and adds
  ~150 lines of Terraform that have nothing to do with the backend
  itself. For a single-instance deployment, the default VPC's
  per-AZ public subnets are exactly what we need. Locking down the
  inbound surface is the SG's job, not the subnet's — and the
  RDS SG only allows ingress from the EC2 SG, which is the
  guardrail that matters.

**RDS for Postgres, self-hosted Redis in Docker on the EC2.**

- Decision: managed RDS for the durable Postgres, a `redis:7-alpine`
  container alongside the app for the ephemeral Redis.
- Alternatives: ElastiCache for Redis (managed, $13/mo minimum, **not
  in Free Tier**); both self-hosted on the EC2 (saves Free Tier
  budget for Postgres, but loses RDS's automated backups and
  point-in-time recovery).
- Why: Postgres holds the durable history of completed games and we
  want managed backups + point-in-time recovery; that maps to RDS.
  Redis holds the ephemeral active-room and active-game state, which
  is intentionally fragile (a Redis restart loses in-flight games,
  documented in `docs/architecture.md`) — paying $13/month for
  managed ephemera is the wrong trade-off. Self-hosted Redis on the
  same EC2 saves ~$15/mo and is fine for portfolio scale.

**Manual first deploy, then automate.**

- Decision: this feature documents a 10-step manual deploy in
  `docs/deploy-runbook.md`. Feature 7.7 replaces steps 3 through 9
  with a GitHub Actions workflow.
- Alternatives: skip the runbook, jump straight to CI/CD;
  semi-automate now with a Bash wrapper.
- Why: see the "manual deploy before automation" concept above. The
  runbook is also the artifact that documents intent — when CI
  fails six months from now, the runbook tells the operator what
  the workflow is trying to do.

**Single-AZ RDS.**

- Decision: `multi_az = false`, single-AZ Postgres.
- Alternatives: Multi-AZ for automatic failover.
- Why: Multi-AZ doubles the cost and is not in the Free Tier. The
  portfolio scope does not need HA Postgres — a chess backend
  outage of a few minutes during an RDS failover is fine.

**`127.0.0.1:8080:8080` binding in the production compose.**

- Decision: bind the app container's port 8080 to the EC2 loopback
  interface only.
- Alternatives: bind to `0.0.0.0:8080`; expose `8080` on the SG.
- Why: Caddy lives on the EC2 host (not in a container), so it
  reaches the app at `localhost:8080` regardless of binding mode.
  Restricting the bind to `127.0.0.1` means even if someone
  accidentally opens port 8080 in the Security Group, the host
  refuses connections from outside. Defence in depth at zero cost.

## How this compares to what I know

- **In Scala / Typelevel, this would be sbt-native-packager + AWS
  CDK (Scala bindings) or Pulumi.** Both are programming-language
  IaC — you write Scala (or Kotlin / TypeScript) and the framework
  emits CloudFormation under the hood. The trade-off vs Terraform
  HCL: a real language means real abstractions (loops, conditionals,
  helper functions) but it also means real complexity. Terraform's
  HCL is dumber on purpose — `for_each` and `count` are the only
  loops, and most production stacks are dataclass-style flat lists
  of resources. The dumb-on-purpose ceiling becomes a feature when
  you are paged at 3 AM and need to read the file fast.

- **In Node, this would be Serverless Framework or AWS CDK.**
  Serverless Framework is a higher-level abstraction targeting
  Lambda + API Gateway; CDK is the same idea as the Scala parallel
  above. The Node ecosystem also has Pulumi, Terragrunt-wrappers,
  and so on. The shape is the same: declare resources, let the
  framework reconcile to the cloud. Terraform's edge over the
  language-IaC tools is the multi-cloud story (the same engine
  drives AWS, GCP, Azure, Cloudflare, Datadog, GitHub) and the
  state-file model that makes drift detection cheap.

- **cloud-init is to a cloud VM what `Dockerfile`'s RUN steps are
  to an image.** Both are "lay down configuration on first
  materialisation"; the difference is that a Docker image bakes the
  layers at build time and is then immutable, while cloud-init
  configures the VM at first boot and the VM is then mutable. For
  a stack like this one, where the VM is treated as
  cattle-not-pets (the runbook says "if something is wrong, destroy
  and re-apply"), the distinction is academic — but the parallel
  helps when you are looking at a cloud-init script for the first
  time and wondering "where does this run?".

- **`@ServiceConnection` from feature 7 is the test-side mirror of
  cloud-init's env-var injection.** In tests, Spring Boot's
  Testcontainers integration discovers the running container's URL
  and rewrites `spring.datasource.url` to point at it. In
  production, cloud-init writes `/opt/chess/.env`, Docker Compose
  loads it, the container reads the env vars, and Spring Boot's
  property resolution rewrites the same key with the RDS endpoint.
  Three contexts, one image, three different mechanisms feeding the
  same env-var slot.

## Gotchas / things I learned the hard way

- **Terraform variables with `sensitive = true` still print to plan
  output if you reference them inside a `local`.** The
  cloud-init template interpolates `deploy_ssh_public_key`. If I
  had referenced `db_password` inside a `templatefile()` call, the
  plan would have surfaced it (Terraform considers `locals` derived
  from sensitive variables sensitive too, so it redacts them — but
  any *string-concatenated* derivation can leak the value into
  resource arguments). Keeping secrets exclusively in the
  `aws_db_instance.password` slot (which Terraform redacts) is the
  safe pattern.

- **AMI lookup needs the right `architecture` filter or you get an
  ARM image.** Canonical publishes both `x86_64` and `arm64`
  Ubuntu 24.04 AMIs under similar names. Without the `architecture
  = x86_64` filter, `data.aws_ami.ubuntu` could resolve to an
  `arm64` image and the `t3.micro` instance type (x86_64) would
  fail to launch. The four filters in `main.tf` (name + virt-type
  + architecture + root-device-type) are the canonical recipe.

- **`aws_eip` charges if unattached.** AWS gives one free Elastic
  IP per region — but only **while it is attached to a running
  instance**. If you stop the EC2 (not terminate, just stop), the
  EIP becomes a billable unattached resource. The Free Tier
  documentation buries this in a footnote. The `aws_eip_association`
  resource is what keeps the attachment durable across Terraform
  runs.

- **`terraform plan` without AWS credentials fails at the AMI data
  source.** The plan needs to query the AMI catalogue to resolve
  `data.aws_ami.ubuntu.id`. Running `terraform plan` against a
  stub `terraform.tfvars` without exported AWS credentials returns
  an `InvalidClientTokenId` from the AMI lookup. The runbook
  documents this — `terraform plan` is part of the real-creds
  workflow, not the offline-validation workflow.

- **`docker compose build` names the image after the directory, not
  the project.** The image produced by `docker compose build` in
  this repo is `chess-backend-java-app`, derived from the directory
  name + service name. The production compose file expects
  `chess-backend:latest`. The runbook includes the re-tag step
  explicitly because forgetting it was my first run-through
  mistake.

## Post-review amendments

Two issues surfaced after the first runbook execution and the first
live `https://` smoke test. Both fixes landed before reviewer
approval and are captured here so the runbook + note + tests stay in
sync.

- **`gzip` not `bzip2` for the `docker save | ssh` image transfer.**
  The runbook originally used `docker save … | bzip2 | ssh … 'bunzip2
  | sudo docker load'`. Ubuntu 24.04 cloud AMI ships with `gzip`
  preinstalled but **not** `bzip2`, and cloud-init does not install
  it — so the receiving end failed with `bunzip2: command not found`.
  Two paths forward: add `bzip2` to the cloud-init `apt install` list,
  or swap both ends to `gzip`/`gunzip`. The second wins: every Linux
  has `gzip`, the compressed image is only ~20 MB larger, and
  removing a dependency beats adding one. The runbook (step 6) and
  this note now use `gzip`/`gunzip`.

- **`server.forward-headers-strategy: framework` is non-optional
  when running behind a TLS-terminating reverse proxy.** Caddy
  terminates TLS on port 443 and proxies plaintext HTTP to
  `localhost:8080`. Without the strategy set, Springdoc inspects the
  inner request — scheme `http`, host `localhost` — and reports
  `servers[0].url = http://chess-backend.duckdns.org` in
  `/v3/api-docs`. The Swagger UI page itself loads over HTTPS, so the
  browser blocks every "Try it out" request to the http URL as
  **mixed content** and the OpenAPI explorer is unusable in
  production. Spring's [`ForwardedHeaderFilter`](
  https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/web/filter/ForwardedHeaderFilter.html)
  fixes this: when `server.forward-headers-strategy=framework` is
  set, the filter consults `X-Forwarded-Proto` and `X-Forwarded-Host`
  (which Caddy sends by default) and rewrites the request URL
  **before** Springdoc reads it. Local `mvn spring-boot:run` is
  unaffected because no forwarded headers are present and the filter
  falls through to the actual request scheme. The regression is
  pinned by `OpenApiForwardedHeadersIT`: one test asserts that
  `servers[0].url` starts with `https://` when `X-Forwarded-Proto:
  https` is sent, the other asserts the local-dev fallback to
  `http://` when no header is present (drift canary — if anyone
  later removes the strategy, the https assertion fails). The
  alternative — `forward-headers-strategy: native` — would defer to
  the servlet container's parser (Tomcat's `RemoteIpValve`),
  which works but couples the behavior to a specific container.
  `framework` keeps the filter in Spring's own code, which is the
  recommended default in Spring Boot 3.x.

## To dig deeper

- [Terraform Language documentation](https://developer.hashicorp.com/terraform/language)
  — the canonical reference; sections on resources, variables,
  outputs, and `templatefile()` are the immediately useful ones.
- [Terraform AWS Provider](https://registry.terraform.io/providers/hashicorp/aws/latest/docs)
  — every resource we declared in `main.tf` has a doc page with all
  its arguments and attributes.
- [AWS Free Tier details](https://aws.amazon.com/free/) — the
  per-service limits. The footnotes (Multi-AZ doubles cost, EIP
  charges when unattached, RDS storage caps at 20 GB) matter more
  than the headline numbers.
- [Caddy documentation](https://caddyserver.com/docs/) — start with
  "automatic HTTPS" and "the Caddyfile". The reverse-proxy reference
  is one page.
- [cloud-init documentation](https://cloudinit.readthedocs.io/) —
  the "examples" page and the "boot stages" page are the two you
  reach for.
- [Let's Encrypt — challenge types](https://letsencrypt.org/docs/challenge-types/)
  — HTTP-01 vs DNS-01; understanding the difference clarifies why
  EC2 SG inbound 80 is non-negotiable for our setup.
- [Duck DNS](https://www.duckdns.org) — the project itself; the UI
  is rough but the service is free and reliable.

New files:

- `infra/versions.tf` — Terraform version pin (>= 1.6) and AWS
  provider pin (~> 5.70). Default tags applied to every resource.
- `infra/variables.tf` — input knobs (region, db credentials, SSH
  keys, allowed CIDR, Duck DNS subdomain, instance/storage sizes,
  budget threshold).
- `infra/outputs.tf` — Elastic IP, RDS endpoint, ECR repo URL,
  convenience SSH commands, Duck DNS FQDN.
- `infra/main.tf` — all resources: VPC + AMI data sources, two
  SGs (EC2 + RDS), key pair, EC2 instance, EIP + association, RDS
  subnet group + instance, ECR repository + lifecycle policy,
  monthly Budget with three notification thresholds.

  Duck DNS subdomain.

  and enables the systemd units.
