# Feature 01 ā€” Health endpoint

**Feature ID:** `health-check`

**Status:** done

---

## What we built

A single GET endpoint, `/api/health`, that returns a JSON document with
`status`, `version`, and `timestamp`. The `version` comes from the build
metadata generated by the Spring Boot Maven plugin; the `timestamp` is
produced through an injected `Clock` so tests can pin time. The endpoint
exists to validate the harness end to end (build, Spring Boot startup,
controller registration, Jackson serialization, Testcontainers IT
pipeline) before any real feature ships.

## Java / Spring concepts that appear

- **`@RestController` + `@RequestMapping` / `@GetMapping`** ā€” Spring scans
  the classpath at startup, finds the controller, and asks
  `RequestMappingHandlerMapping` to bind `GET /api/health` to
  `HealthController#health` by reflection. The routing table is built once
  during context refresh and consulted on every request. Reference:
  [Spring MVC ā€” handler methods](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-methods.html).
- **Constructor injection** ā€” `HealthController` declares its two
  collaborators (`Clock`, `ObjectProvider<BuildProperties>`) as
  constructor parameters and stores them in `final` fields. There is no
  `@Autowired` on fields, per
  [Spring's own recommendation](https://docs.spring.io/spring-framework/reference/core/beans/dependencies/factory-collaborators.html#beans-constructor-injection)
  and per `docs/conventions.md`.
- **`ObjectProvider<BuildProperties>`** ā€” `BuildProperties` is registered
  by Spring Boot's auto-configuration only when
  `META-INF/build-info.properties` is present on the classpath. To avoid
  failing startup when running from an IDE without packaging, the
  controller takes `ObjectProvider<BuildProperties>` (a lazy, optional
  view of the bean) and calls `getIfAvailable()`. This is the
  constructor-injection-friendly alternative to
  `@Autowired(required = false)` on a field. Reference:
  [`ObjectProvider`](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/beans/factory/ObjectProvider.html).
- **`spring-boot-maven-plugin:build-info`** ā€” adding a `<goal>build-info</goal>`
  execution to the Spring Boot Maven plugin tells it to write
  `target/classes/META-INF/build-info.properties` at build time. At
  runtime Spring Boot's `ProjectInfoAutoConfiguration` reads that file and
  binds it into a `BuildProperties` POJO that any bean can inject.
  Reference:
  [Spring Boot Maven plugin ā€” build-info goal](https://docs.spring.io/spring-boot/maven-plugin/build-info.html).
- **`Clock` as an injected dependency** ā€” instead of calling
  `Instant.now()` directly, the controller takes a `java.time.Clock`
  bean (default `Clock.systemUTC()`) and calls `Instant.now(clock)`. Tests
  replace this bean with `Clock.fixed(...)` and assert exact timestamps.
  Reference:
  [`java.time.Clock`](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/time/Clock.html).
- **Records as response DTOs + Jackson** ā€” `HealthResponse` is a
  `record`. Jackson serializes records via their accessor methods (no
  setters needed); Spring picks `MappingJackson2HttpMessageConverter`
  from the configured `HttpMessageConverters` and writes the body. The
  `Instant` field is serialized as an ISO-8601 string thanks to the
  `jackson-datatype-jsr310` module that Spring Boot Web pulls in by
  default. Reference:
  [Jackson record support](https://github.com/FasterXML/jackson-databind/wiki/Jackson-Release-2.12#record-types).
- **`@SpringBootTest` + `MockMvc`** ā€” boots the full Spring context (the
  same one production runs) and dispatches HTTP requests through
  `MockMvc`, which is the `DispatcherServlet` driven by an in-memory
  request/response pair. No real port is opened, but every bean in the
  context is real, so this is closer to an integration test than a slice
  test. Reference:
  [Spring Boot testing](https://docs.spring.io/spring-boot/reference/testing/spring-boot-applications.html).
- **`@TestConfiguration` + `@Primary`** ā€” the nested test class loads an
  extra `@TestConfiguration` whose `@Bean @Primary Clock fixedClock()`
  shadows the production `Clock` bean only for that class. Reference:
  [`@TestConfiguration`](https://docs.spring.io/spring-boot/api/java/org/springframework/boot/test/context/TestConfiguration.html).
- **YAML configuration** ā€” `application.yml` replaces `application.properties`.
  Both formats are valid and Spring Boot reads either; YAML wins on
  nested keys (no prefix repetition). It is parsed via SnakeYAML, which
  Spring Boot includes transitively through `spring-boot-starter`. The
  switch is a repo convention from `docs/conventions.md`.

## Decisions taken

**Dedicated `/api/health` instead of routing through Actuator.**

- Decision: ship a hand-written `HealthController` under `/api/health`.
- Alternatives: expose Actuator's `/actuator/health` directly, or mount
  Actuator under `/api` and customize its response shape.
- Why: the acceptance contract is *exactly* `{ status, version,
  timestamp }`. Actuator's `HealthEndpoint` returns a different shape and
  composes contributions from `HealthIndicator` beans (Postgres, Redis,
  ...). For a first slice, owning the controller is simpler and the
  contract is explicit. Actuator stays mounted under `/actuator` for ops
  use; that is the right home for liveness/readiness probes once we add
  them.

**`ObjectProvider<BuildProperties>` instead of `Optional<BuildProperties>`
or `@Autowired(required = false)`.**

- Decision: declare the dependency as `ObjectProvider<BuildProperties>` in
  the constructor and call `getIfAvailable()` at request time.
- Alternatives: `Optional<BuildProperties>` (Spring 6 supports it for
  constructor params), or a field annotated
  `@Autowired(required = false)`.
- Why: `@Autowired` on fields is banned by `docs/conventions.md`.
  `Optional<BuildProperties>` works but resolves the bean (or its
  absence) at construction time; `ObjectProvider` resolves lazily per
  call, which is the documented Spring idiom for optional dependencies
  in constructors and reads more clearly to anyone familiar with
  Spring.

**Test 3 uses a nested `@TestConfiguration` with `@Primary` instead of
`@MockitoBean`.**

- Decision: nest a small `FixedClockTestConfiguration` inside the IT and
  load it via `@Import` on a `@Nested` class.
- Alternatives: `@MockitoBean Clock clock` plus `when(clock.instant())...`,
  or a top-level test config file under `src/test/java/.../testsupport/`.
- Why: the production code uses `Instant.now(clock)`, which delegates to
  `Clock#instant()` and `Clock#getZone()`. Stubbing a `Clock` with
  Mockito requires stubbing multiple methods and risks drifting from
  what the real `Clock.fixed` returns. `Clock.fixed(...)` is the JDK's
  own canonical implementation and behaves the way the production code
  expects without any stubbing. Keeping the config inside the IT
  (instead of a separate file) makes it obvious why the bean exists and
  scopes the override to a single nested class so the other tests keep
  the real clock.

**`maven-failsafe-plugin` declared explicitly in `pom.xml`.**

- Decision: add the plugin (no version, no config ā€” inherits from the
  Spring Boot parent's `pluginManagement`) so `./mvnw verify` actually
  runs the `*IT` tests.
- Alternatives: rename to `*Test` and let surefire run them, or change
  the include pattern in surefire.
- Why: `docs/conventions.md` says `*IT.java` for integration tests, and
  `init.sh` ends with `./mvnw verify`. The starter parent ships
  `pluginManagement` for failsafe but does *not* enable it in `build`.
  Declaring the plugin (one line of XML) is the standard Spring Boot
  way to flip the switch.

## How this compares to what I know

This section is for a reader coming from Scala / Typelevel.

- **Routing.** In `http4s`, the route table is an explicit value:
  ```scala
  val routes = HttpRoutes.of[IO] {
    case GET -> Root / "api" / "health" => Ok(healthResponse)
  }
  ```
  In Spring it is inferred at startup from `@GetMapping("/api/health")`
  on a method via classpath scanning + reflection. The Scala version is
  a value you can `combineK` and refactor like any other value; the
  Spring version is a registration side effect on a singleton bean. The
  surprise from the Typelevel side is that you cannot "see" the route
  table by reading `main` ā€” it materializes at context refresh from
  annotations scattered across classes.

- **Wiring.** Cats Effect's `IOApp` makes the wiring obvious:
  ```scala
  object Main extends IOApp.Simple {
    val run = (mkClock, mkBuildInfo).mapN(HealthController.apply).flatMap(serve)
  }
  ```
  Spring's container does the same job, but by reflection on
  `@Configuration` classes and constructor parameter types at startup.
  Same idea (dependencies are values handed in), different mechanism
  (declarative + reflective vs. explicit + value-level). The surprise:
  swapping a dependency in Spring means swapping the *bean definition*,
  not changing a `val` in `main`.

- **Time.** Cats Effect models time as a capability:
  ```scala
  def now[F[_]: Clock]: F[Instant] = Clock[F].realTimeInstant
  ```
  Java's `java.time.Clock` is the same idea minus the effect type. The
  shape is `clock.instant()` (no `F[_]`), which means there is no
  composition with other effects ā€” but for testability the pattern is
  identical: take a `Clock`, do not call `Instant.now()` directly.

- **Configuration.** `Doobie`/`http4s` apps typically take config via
  `pureconfig` or `ciris` and pass it as a value. Spring binds
  `application.yml` to properties via `Binder` and exposes the bound
  values either as beans (`@ConfigurationProperties`) or as injected
  `@Value`-resolved strings. Same end result; the Scala version is
  value-level and total, the Spring version is reflective and bound at
  context refresh.

- **Tests.** A `cats.effect.testing` test is a free-standing program: you
  build the world, run an IO, assert. `@SpringBootTest` instead boots a
  full container, populated by the same auto-configuration that runs in
  production. The lifecycle is heavier (context cached across tests, but
  re-built when configuration changes ā€” e.g., adding a nested
  `@Import`), so test isolation is a different story than passing fresh
  state into a function.

## Gotchas / things I learned the hard way

- `BuildProperties` is **not** registered unless the Maven plugin's
  `build-info` goal is wired in. The default Spring Boot starter parent
  does not run it. Adding the execution to `spring-boot-maven-plugin`
  was a one-line fix, but without it Spring throws
  `NoSuchBeanDefinitionException` at the controller constructor.
- `TestcontainersConfiguration` shipped as package-private. That works
  if every IT sits in the root `chess` package, but `web/health/`
  immediately broke compilation. Making the class and its `@Bean`
  methods `public` is the right fix ā€” ITs live in sub-packages by
  feature.
- The Spring Boot parent declares `maven-failsafe-plugin` under
  `<pluginManagement>` but does **not** enable it in `<build>`.
  `./mvnw verify` was silently a no-op for `*IT` tests until I added the
  plugin to the project's `<build><plugins>` block. The first run
  reported "all checks passed" with zero IT tests executed ā€” a sneaky
  green.
- `@AutoConfigureMockMvc` lives in
  `org.springframework.boot.test.autoconfigure.web.servlet`, **not**
  under `org.springframework.test.web.servlet.autoconfigure`. The IDE
  guessed wrong and the compile error was the next sign.
- The failsafe `.txt` summary printed `Tests run: 0` even though the XML
  report and the JVM logs both clearly showed three tests executing.
  Trust the XML ā€” `<testsuite tests="3" failures="0"/>` is authoritative.
