# Feature 07.7 — Backend CI/CD pipeline

**Feature ID:** `backend-cicd-pipeline` (from `feature_list.json`)

**Status:** in progress

---

## What we built

Automated deploys to AWS on every push to `main`. A new
`.github/workflows/deploy.yml` runs `./init.sh`, authenticates to AWS via
**OIDC** (no long-lived keys anywhere), builds and tags the Docker image
with the commit SHA + `latest`, pushes both tags to ECR, SSHes into the
EC2 as `deploy` to pull and restart the container via
`docker compose pull && up -d`, and smoke-tests the public
`/api/health` endpoint before reporting green. The Terraform stack in
`infra/` grows two IAM identities: a `github_actions` role assumable
only from this repo's `main` branch via the GitHub OIDC provider, and an
`ec2_ecr_pull` instance profile attached to the running EC2 so the host
can pull from ECR via IMDSv2.

## Java / Spring concepts that appear

This is a CI/CD + AWS feature; the concepts are DevOps and security, not
`@Annotation` machinery. They are still the ones a Spring Boot developer
has to internalize the day they take a service to production.

- **OIDC-federated AWS authentication.** OpenID Connect is a thin layer
  on top of OAuth 2.0 that adds an identity token: a JWT signed by the
  identity provider (here, GitHub) and verifiable by the relying party
  (here, AWS) using the provider's public keys. The handshake in our
  workflow is: (1) `actions/configure-aws-credentials@v4` asks the
  GitHub Actions runtime for a JWT scoped to this job — that is the
  step that `permissions: id-token: write` enables; (2) the action
  posts the JWT to AWS STS via `sts:AssumeRoleWithWebIdentity`; (3) STS
  validates the signature against the OIDC provider we registered in
  Terraform, checks the `aud` claim equals `sts.amazonaws.com`, and
  checks the `sub` claim matches the trust-policy condition; (4) STS
  returns short-lived access keys (typically 1 hour) that the workflow
  uses for the remainder of the job. No static key material is ever
  stored in a secret store. See the [AWS docs on OIDC for GitHub
  Actions](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
  and the [GitHub docs on OIDC](
  https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/about-security-hardening-with-openid-connect).

- **The `sub` claim trust-policy condition.** The
  `token.actions.githubusercontent.com:sub` claim in the JWT looks like
  `repo:dariogguillen/chess-backend-java:ref:refs/heads/main`. Our
  trust policy condition does `StringLike` against the exact string —
  any workflow not running on `main` of this repo gets a
  `Could not assume role` failure from STS. **Without this condition**,
  the role would be assumable from any GitHub Actions workflow in any
  repository in the world that happens to know the role ARN — a
  catastrophic open door. Other useful claim shapes:
  `repo:owner/repo:environment:production` (when you start using GitHub
  Environments), `repo:owner/repo:pull_request` (for PR validation
  workflows). The choice between them is the deploy-pipeline equivalent
  of choosing controller URL patterns: be as specific as the trust
  model demands.

- **`permissions: id-token: write` as the OIDC enable bit.** The GitHub
  Actions runtime mints the OIDC JWT only when a job explicitly opts
  in via this permission. Default is `none`. A job without it can
  still call `configure-aws-credentials`, but the action fails at the
  "request token" step with a confusing 400 error. This is the #1
  OIDC gotcha — the symptom is generic AWS auth failure, but the root
  cause is one missing line at the top of the workflow. The other
  permission, `contents: read`, is required so the job can `checkout`
  the repo; without it, the checkout step also fails opaquely.

- **IAM instance profile + IMDSv2.** An "instance profile" is a thin
  wrapper around an IAM role that EC2 understands; attaching it to
  the `aws_instance` resource tells AWS to deliver credentials for
  that role to the instance via the **Instance Metadata Service v2**.
  When `awscli` or the AWS SDK runs on the EC2, it walks a documented
  search order — env vars, shared credentials file, ECS container
  metadata, EC2 metadata — and hits IMDSv2 last. IMDSv2 uses a
  token-protected request flow that fixes the SSRF vulnerability the
  original IMDSv1 had. The result: no `aws configure`, no key files
  on disk, automatic rotation every few hours, and a tiny least-
  privilege blast radius (`AmazonEC2ContainerRegistryReadOnly` only).
  See the [IMDSv2 documentation](
  https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/configuring-instance-metadata-service.html).

- **`AmazonEC2ContainerRegistryPowerUser`.** AWS-managed policy that
  grants `ecr:*` minus the destructive `ecr:DeleteRepository`,
  `ecr:DeleteLifecyclePolicy`, etc. Convenient for the CI role
  because it covers push + pull + describe. The trade-off is that it
  is **account-wide**: any ECR repo in the account is fair game. A
  least-privilege replacement scoped to the `chess-backend` repo
  would be a `Resource = aws_ecr_repository.app.arn` constraint in a
  custom policy document. Surfaced as polish in "Decisions taken"
  below.

- **Docker image tag strategies.** Two tags pushed per deploy:
  `<commit-sha>` (immutable — the exact build produced by this run)
  and `latest` (mutable — always points at the most recent successful
  deploy). The SHA tag is what the deploy step references in
  `APP_IMAGE`, so the running container is traceable to a specific
  commit; `latest` exists so a human running the manual procedure can
  `docker pull <repo>:latest` without looking up a hash. Rollback is
  `APP_IMAGE=<repo>:<older-sha> docker compose -f docker-compose.prod.yml up -d`.

- **`docker compose pull && up -d` and image-hash-driven recreation.**
  Compose v2 inspects the image manifest at `pull` time and records
  the new digest. The subsequent `up -d` compares each service's
  current container against the new spec; services whose effective
  config (including image hash) is unchanged are left alone, services
  whose hash differs are recreated. In our prod compose, `redis` is
  pinned to `redis:7-alpine` and never changes, so it stays running
  across deploys; only the `app` container restarts. Layer-level
  download dedup makes the `pull` itself fast once the base image is
  cached on the host.

- **GitHub Actions `concurrency`.** Our workflow uses
  `group: deploy-prod, cancel-in-progress: false`. Meaning: at most
  one deploy runs at a time; if a second push lands while a deploy is
  in flight, the second run **queues** and starts after the first
  finishes. The alternative, `cancel-in-progress: true`, would cancel
  the in-flight deploy as soon as the second push arrives — risky
  here because the deploy is non-atomic (build → push → SSH → restart)
  and a half-finished deploy is worse than a slightly stale one. For
  pipelines with cheap, idempotent deploys, the opposite call is
  fine. See the [GitHub docs on
  concurrency](https://docs.github.com/en/actions/using-jobs/using-concurrency).

- **`ssh-keyscan -H` to bootstrap `known_hosts`.** SSH refuses to
  connect to a host whose key is not in `known_hosts` unless the
  user accepts interactively. CI has no human to accept the prompt,
  so the workflow runs `ssh-keyscan -H <host> >> ~/.ssh/known_hosts`
  before the SSH call. `-H` hashes the hostname in the resulting
  entry; a leakier but lazier alternative is `StrictHostKeyChecking=no`.
  More paranoid alternatives: pin the host key as a secret and write
  it directly into `known_hosts`. For a single-host portfolio deploy
  the `keyscan` approach is conventional and adequate.

## Decisions taken

**OIDC over static IAM user keys.**

- Decision: use the GitHub Actions OIDC provider + an IAM role with a
  trust policy scoped to `repo:.../main`. No static `AWS_ACCESS_KEY_ID`
  or `AWS_SECRET_ACCESS_KEY` secrets anywhere.
- Alternatives: create a dedicated IAM user, generate a key pair,
  store both halves as GitHub Actions secrets (the historical pattern,
  still works fine).
- Why: the OIDC pattern eliminates the single largest CI security
  failure mode — leaked static keys that have no expiry and can be
  used from anywhere. Short-lived STS credentials cap the blast radius
  at one job. The setup is one extra Terraform resource (the OIDC
  provider) and a trust policy condition; the runtime difference is
  zero. CV-signal premium: this is the 2026 best practice for AWS +
  GitHub Actions, surfaced in every AWS security talk and most platform
  team interviews.

**`AmazonEC2ContainerRegistryPowerUser` (convenience) on the GHA role.**

- Decision: attach the AWS-managed `PowerUser` policy to the
  `github_actions` role.
- Alternatives: write a custom IAM policy document with the exact
  ECR actions (`ecr:BatchCheckLayerAvailability`,
  `ecr:CompleteLayerUpload`, `ecr:GetAuthorizationToken`,
  `ecr:InitiateLayerUpload`, `ecr:PutImage`, `ecr:UploadLayerPart`)
  scoped via `Resource = aws_ecr_repository.app.arn` (and `*` for
  `GetAuthorizationToken`, which is account-wide).
- Why: the managed policy is the conventional starting point and the
  least-privilege rewrite is a 30-line tangent that does not change
  observable behavior. The trade-off is that any other ECR repo in
  this account is reachable from this CI role — given we have exactly
  one repo and one project on this AWS account, the practical risk is
  zero today. **Polish opportunity**: when a second project shows up
  on this account, swap to a custom policy.

**`AmazonEC2ContainerRegistryReadOnly` on the EC2 instance profile.**

- Decision: attach the AWS-managed read-only policy.
- Alternatives: write a custom least-privilege policy scoped to one
  repository.
- Why: the EC2 only ever pulls. Read-only is already strictly minimum
  on the action axis; scoping further by resource is the same polish
  opportunity as above. The read-only managed policy is account-wide
  on resources but does not grant push, so an attacker who compromised
  the EC2 cannot tamper with images.

**OIDC root thumbprint hardcoded.**

- Decision: hardcode
  `6938fd4d98bab03faadb97b34396831e3780aea1` as the
  `aws_iam_openid_connect_provider.thumbprint_list` value.
- Alternatives: use the Terraform `tls` provider's
  `data.tls_certificate` to fetch the live cert chain and compute
  the thumbprint dynamically at plan time.
- Why: AWS recently switched their OIDC provider validation logic to
  use GitHub's [JWKS](https://token.actions.githubusercontent.com/.well-known/openid-configuration)
  directly and the thumbprint became advisory — see the [AWS
  announcement](https://github.com/aws-actions/configure-aws-credentials/issues/357).
  Hardcoding the historical thumbprint is the documented compatibility
  approach. Dynamic computation via `data.tls_certificate` would work
  too, but adds a new provider dependency and a new failure mode (the
  TLS fetch can fail) for no current benefit. Documented choice; will
  revisit if AWS deprecates the field entirely.

**Dedicated CI SSH key, not the user's personal key.**

- Decision: the runbook (section 12) generates a new `~/.ssh/chess_ci`
  ed25519 keypair, appends the public half to `deploy@<eip>:.ssh/authorized_keys`,
  and stores the private half as `DEPLOY_SSH_KEY` in GitHub Secrets.
- Alternatives: reuse the user's personal `~/.ssh/id_ed25519` for
  both interactive access and CI.
- Why: rotation, audit, blast radius. The user can revoke the personal
  key after a laptop loss without breaking CI; the user can revoke the
  CI key after a workflow review without breaking interactive access.
  The cost is one minute of setup and one extra entry in
  `authorized_keys`.

**`docker compose pull && up -d` over `docker save | ssh | docker load`.**

- Decision: registry-native pulls via ECR.
- Alternatives: keep the 7.5 manual procedure's `docker save | ssh |
  docker load` flow.
- Why: ECR exists; we already pay nothing for it (Free Tier). Pulling
  from ECR is faster than streaming a tarball over a home connection,
  produces an audit trail of pushed images, and uses the same
  `docker compose pull` mechanism a non-CI deploy would. The
  `docker save` flow stays in the runbook as a fallback when CI is
  unavailable.

**Concurrency `cancel-in-progress: false` (queue, do not drop).**

- Decision: rapid main pushes queue.
- Alternatives: cancel the in-flight deploy when a new push arrives
  (`cancel-in-progress: true`).
- Why: the deploy is non-atomic — between `docker push` and
  `docker compose up -d` there is a window where the new image is in
  ECR but not yet running. Cancelling mid-flight risks leaving ECR
  with a new image and the EC2 with a partial restart. Queuing is
  safer for portfolio cadence where the cost is a few extra seconds.

**No staging environment.**

- Decision: only production. Pushes to `main` go straight live.
- Alternatives: a `staging` branch deploying to a parallel
  Terraform stack.
- Why: the portfolio scope does not justify the second monthly
  Free Tier instance and the doubled `terraform apply` discipline.
  When this graduates to a paying side project with real users, the
  next move is to add a `staging` branch + a `develop` environment
  GitHub Environment + a Terraform `workspace`; out of scope today.

**`workflow_dispatch` enabled.**

- Decision: the workflow has both `push` and `workflow_dispatch`
  triggers.
- Alternatives: push-only.
- Why: `workflow_dispatch` adds a "Run workflow" button in the
  Actions UI — useful for re-running a failed deploy without an empty
  commit, and for testing the workflow itself without polluting git
  history. Zero downside.

## How this compares to what I know

- **In Scala / Typelevel.** The closest parallel is using sbt with a
  release plugin like `sbt-release` or `sbt-ci-release`, driven by a
  similar GitHub Actions workflow. The OIDC + AWS layer is identical
  — `aws-actions/configure-aws-credentials` is JVM-agnostic. The
  build step (here `./init.sh` invoking `./mvnw verify`) would be
  `sbt clean test`, and the image push step would be
  `sbt Docker/publish` against an ECR endpoint after `docker login`.
  The keyless-IAM pattern is one of the few CI/CD primitives that
  ports straight across ecosystems with zero adaptation.

- **In Node.** Same shape with `npm ci && npm test && docker build`
  swapped in for the Java steps. The closest production-grade
  template is the
  [official AWS sample](https://github.com/aws-actions/configure-aws-credentials/blob/main/.github/workflows/test.yml).
  GitHub Actions itself is Node-friendly because composite actions
  (the action layer the marketplace runs on) is largely Node — but
  none of that matters to the OIDC + ECR + SSH chain.

- **`AmazonEC2ContainerRegistryPowerUser` vs the AWS Free Tier
  trade-off.** Comparable to using `:latest` Docker base images in
  early development versus pinning to a specific digest the moment
  you go to production. Convenience policies and convenience tags
  are the same shape of decision: zero added complexity today, a
  documented "swap when the scale changes" path.

- **OIDC trust policy `sub` claim vs Spring Security's `@PreAuthorize`.**
  Both are declarative authorization decisions enforced at the boundary.
  Spring's `@PreAuthorize("hasRole('ADMIN')")` and AWS's
  `token.actions.githubusercontent.com:sub = repo:owner/repo:ref:refs/heads/main`
  are conceptually the same — a string predicate over a request's claims
  that decides whether to admit the call. The difference is the
  enforcement point: Spring evaluates inside the JVM after the request
  hits the controller; AWS evaluates inside STS before any AWS API is
  reachable.

- **IMDSv2 vs the JVM Service Provider Interface.** Both are
  "ambient context the runtime exposes that you do not have to
  configure". A Spring `@Service` injected via constructor is what
  `AWSCredentialsProviderChain` resolves to on an EC2 with an
  instance profile — the consumer asks for credentials, the chain
  walks its sources, the IMDSv2 source answers without you having
  written `aws configure` anywhere.

## Gotchas / things I learned the hard way

- **The `iam_instance_profile` attribute is in-place mutable in
  Terraform.** Adding it to an existing `aws_instance` shows `~`
  in the plan, not `-/+`. This is critical — replacing the running
  EC2 would lose `/opt/chess/.env`, the Docker images on disk, and
  any in-flight game state in Redis. The
  [provider docs](https://registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/instance#iam_instance_profile)
  document the attribute as mutable; the plan output is the proof.

- **The OIDC root thumbprint is a moving target.** AWS originally
  required the thumbprint to match GitHub's certificate exactly;
  rotation events broke pipelines that hardcoded an outdated value.
  Modern AWS validates the JWT signature directly via GitHub's JWKS,
  making the thumbprint advisory — but `aws_iam_openid_connect_provider`
  still requires the field, so the hardcoded historical value remains
  the path of least resistance. See [aws-actions issue #357](
  https://github.com/aws-actions/configure-aws-credentials/issues/357).

- **Cloud-init runs only on first boot.** Updating
  `infra/ec2-cloud-init.sh.tpl` to install the AWS CLI does **nothing**
  to the existing instance because `lifecycle { ignore_changes =
  [user_data] }` is set and cloud-init does not re-run on
  user_data changes anyway. The fix for the running EC2 is a
  one-shot manual install documented in section 11 of the
  runbook. Future `terraform destroy + apply` cycles will produce
  instances with the AWS CLI pre-installed.

- **Ubuntu 24.04 dropped the `awscli` apt package.** AWS no longer
  distributes CLI v1 via apt, and v2 is shipped exclusively through
  the official installer zip at `awscli.amazonaws.com/awscli-exe-linux-x86_64.zip`.
  Cloud-init therefore installs `unzip` (not preinstalled on the
  Ubuntu cloud AMI), downloads the bundle, and runs `/tmp/aws/install`,
  landing the binary at `/usr/local/bin/aws`. The runbook's one-shot
  upgrade in section 11 follows the same pattern. Trying
  `apt-get install -y awscli` on 24.04 fails with
  `Package 'awscli' has no installation candidate` — the symptom
  surfaced once a fresh AMI was booted.

- **`docker compose pull` ignores the registry when the image tag
  resolves to a local image with the same name.** The
  `${APP_IMAGE:-chess-backend:latest}` default lets the manual
  procedure work locally without a registry, but the CI workflow
  must always set `APP_IMAGE` to the full ECR URL or `pull` will
  silently skip the registry round-trip. The workflow does this
  explicitly: `APP_IMAGE="${ECR_REPO}:${IMAGE_TAG}"` is passed on
  the same SSH command line as the `docker compose pull` and
  `up -d` calls.

- **Missing trailing newline on a multi-line GitHub Actions secret
  breaks SSH.** Pasting the contents of `~/.ssh/chess_ci` into the
  web UI is fine if you preserve the trailing `\n` after
  `-----END OPENSSH PRIVATE KEY-----`. Most browsers strip it.
  Symptom: the workflow's `ssh -i ~/.ssh/deploy_key` step exits
  with `Load key "/root/.ssh/deploy_key": error in libcrypto`.
  The `gh secret set DEPLOY_SSH_KEY < ~/.ssh/chess_ci` path avoids
  this by reading the file bytes verbatim — the runbook surfaces
  this as a known cause if the web UI path is taken.

- **`permissions: id-token: write` is the OIDC enable bit and the
  symptom of its absence is a 400 with no useful message.** When I
  first wrote this workflow I forgot the line and saw
  `Error: Could not load credentials from any providers. (Error
  fetching JWT)`. The fix is one line at the top of the workflow.
  Add it once, never touch it again.
