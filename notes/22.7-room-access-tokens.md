# Feature 22.7 — Separate join token (room access tokens)

**Feature ID:** `room-access-tokens` (from `feature_list.json`)

**Status:** done

---

## What we built

Until now the six-character `roomId` was the single shared secret:
`POST /api/rooms/{id}/join` needed only the `roomId`, so anyone the
creator shared the room with for spectating (now first-class after
feature 22.5) could occupy the second-player slot. This feature
separates the **capability to join** from the **capability to watch**: a
secret `joinToken` is minted at room creation, returned only in the
create response, and required by the join endpoint. A missing or wrong
token returns `403 INVALID_JOIN_TOKEN`. The `roomId` stays the
public-ish identifier for watching (`GET /api/rooms/{id}` and
`SUBSCRIBE /topic/rooms/{roomId}`) and no longer authorises joining.

## Java / Spring concepts that appear

- **Record evolution with a backwards-compatible canonical
  constructor** — `Room` gained a sixth component, `String joinToken`,
  appended last in the canonical constructor. The three older
  convenience constructors delegate with `joinToken = null`, and the
  compact constructor simply does not validate it (null is allowed).
  This is the same pattern the project already used for `creatorSide`
  (feature 21) and `timeControl` (feature 22). The reason it matters:
  Jackson deserialises a record by calling its canonical constructor,
  and a JSON document persisted in Redis *before* the field existed has
  no `joinToken` key — Jackson passes `null` for the missing component,
  and the room comes back as a legacy/unprotected room. See the Java
  language spec on
  [record declarations](https://docs.oracle.com/javase/specs/jls/se21/html/jls-8.html#jls-8.10).
- **A narrow `@ExceptionHandler` on a concrete exception type** —
  `InvalidJoinTokenException extends ChessException` directly (no
  `ForbiddenException` umbrella), and `GlobalExceptionHandler` maps it
  with `@ExceptionHandler(InvalidJoinTokenException.class)` to
  `HttpStatus.FORBIDDEN`. This mirrors how
  `InvalidCredentialsException` is handled for 401. The error *code*
  (`INVALID_JOIN_TOKEN`) is produced by the existing mechanical
  `codeOf(ChessException)` — simple class name minus `Exception`,
  camelCase → UPPER_SNAKE_CASE — so no new branch was needed there.
  [Spring `@ExceptionHandler` docs](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-exceptionhandler.html).
- **`@Schema(allowableValues = …)` as a spec-level enum + the OpenApiIT
  canary** — adding a code is a two-sided edit: the new literal goes
  into `ErrorResponse.error`'s `allowableValues` (springdoc renders it
  as an enum array that `openapi-typescript` turns into a TS union on
  the frontend), and `OpenApiIT` asserts the spec enum equals an exact
  expected set. The canary fails the build if you update one half and
  forget the other — the discipline feature 6.6 introduced.
- **`UUID.randomUUID()` as an unguessable bearer secret** — 122 bits of
  randomness, generated server-side. Contrast the `roomId`, a 6-char
  code from a 32-symbol alphabet (~30 bits) that is deliberately short
  and shareable: short enough to be guessable, which is exactly why it
  must not double as the authorisation secret.

## Decisions taken

- **Decision:** the token lives on the `Room` (persisted in Redis) as a
  nullable `String`, with `null` meaning "legacy / unprotected room".
  **Alternatives considered:** a separate `room-token:{id}` Redis key;
  or requiring a token unconditionally (no null escape hatch).
  **Why this one:** keeping it on the aggregate is the smallest change
  and rides the existing record-evolution pattern. The null-as-legacy
  rule is the deploy-safety hinge — rooms already in Redis when the
  deploy lands keep accepting token-less joins until they TTL out (24h),
  so live games are not broken mid-session. A hard requirement would
  have 403'd every in-flight room instantly.
- **Decision:** transport the token in the request **body**
  (`JoinRoomRequest.joinToken`), not a header. **Alternatives:** an
  `Authorization`-style or custom header. **Why:** the endpoint already
  takes a JSON body (`displayName`); adding a field is consistent and
  needs no CORS allow-list change (a custom header would — see the
  feature 11.7 `X-Player-Id` saga).
- **Decision:** validation order inside the atomic block is
  room-exists → token-valid → not-full. **Why:** a caller without the
  token gets 403 even on a full room, so we never leak room state to
  someone who cannot join.
- **Decision:** `joinToken` is **not** `@NotBlank` on the request.
  **Why:** whether a token is required is a *business* decision made in
  the service (a legacy room accepts a token-less join), so a missing
  token must surface as 403 from the service, not as a 400 validation
  failure at the boundary.
- **Decision:** the token is returned **only** on the create response;
  the join response sets it `null` and `GET /api/rooms/{id}` has no
  token field at all. **Why:** the creator is the only party that needs
  it; the joiner is already in, and the watch path must never leak it.

## How this compares to what I know

- **In Scala / Typelevel this would be…** a **capability URL / bearer
  token** — possession equals permission, decoupled from identity. I'd
  model the token as an opaque newtype rather than a bare `String` —
  `final case class JoinToken(value: String) extends AnyVal` (or a
  `@newtype`) — so it can't be confused with the `roomId` or a
  `displayName` at a call site. The room would carry
  `Option[JoinToken]`, and `None` would encode the same legacy-room
  meaning that `null` does here (Java's `null` is doing the job of
  `Option` — the record-evolution comment is explicit that null is a
  *meaningful* value, not an accident). The join check is naturally an
  `Either[JoinError, Unit]` or a `ValidatedNel` at the boundary:
  `existing.joinToken match { case None => Right(()); case Some(t) =>
  Either.cond(provided.contains(t), (), InvalidJoinToken) }`. Spring's
  `@ExceptionHandler` → HTTP status mapping is the imperative analogue
  of an http4s middleware that turns a `JoinError` into a `Forbidden`
  response.
- **In Node this would be…** a token in the JSON body checked in an
  Express handler, throwing a typed error that an error-handling
  middleware maps to `res.status(403)`. The OpenAPI enum + generated
  client union has a direct analogue if you drive types from a shared
  schema (e.g. `zod` + `zod-to-openapi`), which is the same
  single-source-of-truth idea the `@Schema(allowableValues)` + canary
  enforces here.

## Gotchas / things I learned the hard way

- The bulk of the diff was **not** the feature — it was the mechanical
  sweep of every integration-test helper that does create-then-join.
  Adding the token requirement turns every old `POST /join` into a 403,
  so ~8 ITs (`RoomControllerIT`, `RoomDetailsControllerIT`,
  `GameControllerIT`, `MyGamesIT`, `ViewerCountIT`, `RoomLifecycleIT`,
  `TimeControlIT`, `DisconnectHandlingIT`, `DisconnectNotificationsIT`,
  `GameWebSocketIT`, `StompAuthIT`) had to read `joinToken` from the
  create response and thread it onto the join body. `./init.sh` fails
  loudly on any you miss, which is the safety net working as intended.
- The legacy-room (null-token) path **cannot** be produced through the
  new REST API (create always stamps a token), so it can only be proven
  at the service layer — hence the dedicated `RoomServiceTest` case that
  builds a `Room` via the 4-arg convenience constructor (joinToken
  defaults to null) and asserts a token-less join succeeds.
