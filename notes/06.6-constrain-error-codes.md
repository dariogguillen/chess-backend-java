# Feature 06.6 ŌĆö Constrain error codes in the OpenAPI spec

**Feature ID:** `constrain-error-codes` (from `feature_list.json`)

**Status:** in progress

---

## What we built

Annotated `ErrorResponse.error` with `@Schema(allowableValues = {...})`
listing the nine error codes that `GlobalExceptionHandler` actually
emits today. Springdoc renders this as an `enum` array on the OpenAPI
spec, so `chess-frontend`'s codegen via `openapi-typescript` will narrow
the field to a TypeScript union literal automatically ŌĆö no parallel
hand-maintained list on the client. A drift-canary IT in `OpenApiIT`
pins the exact set, and `docs/architecture.md`'s API contract section
gains a codes table that is now the cross-repo source of truth.

## Java / Spring concepts that appear

- **`@Schema(allowableValues = {...})`** ŌĆö Swagger-core annotation that
  Springdoc picks up during component scanning and translates to OpenAPI's
  `enum` field on the property schema. The values are written as String
  literals in the annotation, in source order; Springdoc preserves that
  order in the rendered spec. The alternative is
  `@Schema(implementation = SomeEnum.class)` when you have an actual
  Java `enum` ŌĆö Springdoc then reflects the enum constants and writes
  them out. We deliberately chose the String-literal form (see Decisions).
  See the [Springdoc reference for OpenAPI 3
  generation](https://springdoc.org/#features) and the
  [Swagger `@Schema`
  Javadoc](https://docs.swagger.io/swagger-core/v2.2.7/apidocs/io/swagger/v3/oas/annotations/media/Schema.html).

- **Spec-driven type safety across language boundaries.** OpenAPI is the
  serialized contract; `openapi-typescript` is a codegen that consumes
  it and produces TypeScript types. When the backend declares
  `enum: [...]` on a `string` property, the codegen turns the property
  into a union literal (`"ROOM_NOT_FOUND" | "ROOM_FULL" | ŌĆ”`) instead
  of `string`. The static guarantee is "the type on the client cannot
  drift from the contract on the server because the type is generated
  from the contract" ŌĆö a one-way enforcement. The other direction
  (runtime: server returning a code outside the enum) is still
  possible if our handler emits something it shouldn't, which is what
  the drift canary protects against.

- **Mechanical `codeOf` vs hardcoded literals ŌĆö an intentional
  asymmetry.** `GlobalExceptionHandler`'s typed-exception handlers
  (`handleNotFound`, `handleConflict`, `handleUnprocessable`) all delegate
  to `codeOf(ChessException)`, which strips the trailing `Exception` and
  converts camelCase to UPPER_SNAKE_CASE. Add a new
  `SomethingException extends NotFoundException` and you get
  `SOMETHING` for free ŌĆö no new branch in the handler. The other three
  handlers (`MethodArgumentNotValidException`, `HttpMessageNotReadableException`,
  `MissingRequestHeaderException`) target Spring-framework exceptions
  whose names do not follow our convention, so their codes are
  hardcoded literals (`VALIDATION_FAILED`, `MALFORMED_REQUEST`,
  `MISSING_HEADER`). This asymmetry is documented in the architecture
  doc; adding a new framework handler means adding a new literal.

- **Drift canary as a test pattern.** Whenever an internal set is
  exposed through an external surface ŌĆö error codes here, allowed
  origins, log levels, feature flags ŌĆö there is a risk that one half
  drifts from the other. The canary pins the exact set in a test and
  fails the build on any change, forcing the developer to update both
  halves together. The test deliberately uses a sort-then-compare so
  it does not couple to source order, and it covers two failure
  modes: missing annotation entirely (no `enum` field in the spec)
  and set mismatch (annotation drift from the handler).

## Decisions taken

**Nine codes today, not aspirational ones.**

- Decision: enumerate exactly the nine codes that
  `GlobalExceptionHandler` emits today.
- Alternatives: include the full forward-looking set from
  `docs/conventions.md`'s exception-hierarchy diagram (which lists
  exception classes that have no live subclasses yet); or include all
  codes that have ever existed.
- Why: the spec is a contract about live behavior, not a wish list.
  An aspirational code in the enum would let a frontend match on it
  in a `switch` whose case is unreachable until a backend feature
  ships that emits it. When the next feature lands, it updates the
  enum at the same time it starts emitting the code ŌĆö the drift
  canary makes that mandatory. `ROOM_ALREADY_STARTED` was removed in
  feature 4 cleanup precisely because its emission path was
  unreachable; we are not putting it back.

**String literals in `@Schema`, not a Java enum.**

- Decision: list the nine codes as String literals in
  `@Schema(allowableValues = {...})`.
- Alternatives: introduce a Java `enum ErrorCode { ROOM_NOT_FOUND,
  ROOM_FULL, ŌĆ” }` and use `@Schema(implementation = ErrorCode.class)`
  on the `error` field; the handler would emit `ErrorCode.ROOM_FULL.name()`.
- Why: the codes already live as String literals in two places ŌĆö the
  derived `codeOf` output for typed exceptions and the hardcoded
  literals in the framework handlers. Introducing a Java enum would
  add a third place to keep in sync and would not eliminate either
  existing one (the `codeOf` derivation cannot return a typed enum
  without losing its "mechanical, no-new-branch" property). The
  annotation is the minimal correct expression. If we ever decide to
  refactor the handler to use a Java enum, the `@Schema(implementation =
  ErrorCode.class)` swap is a one-line change.

**Sort-then-compare in the IT, not order-preserving compare.**

- Decision: the drift canary sorts both `actual` and `expected` before
  comparing.
- Alternatives: compare in source order; rely on Springdoc to
  preserve the order of the `allowableValues` array; assert on a
  `Set` directly.
- Why: source order is an implementation detail of the annotation; a
  future refactor could reorder for readability (alphabetical, by
  HTTP status, by feature) without changing the contract. Pinning
  the order in the test would invite false failures. `Set`
  comparison would lose duplicate detection ŌĆö if `allowableValues`
  accidentally lists `ROOM_FULL` twice, the spec would expose a
  10-element enum with a duplicate, and a Set comparison would still
  pass. `List`-after-sorting catches both reordering robustness and
  duplicate detection in one assertion.

**`docs/architecture.md` table as cross-repo source of truth.**

- Decision: extend the existing "API contract" subsection with a
  table of nine codes + HTTP status + originating exception. The
  frontend's own `docs/architecture.md` mirrors it if and when it
  surfaces the contract directly in its docs.
- Alternatives: put the table in the README; in a new top-level doc;
  in the `notes/` folder; or skip the doc entirely (the spec is the
  source of truth, the prose duplicates it).
- Why: same pattern as the STOMP API contract section ŌĆö backend
  canonical, frontend mirrors when needed. Skipping the doc would be
  defensible (the spec is parseable, the table is human-readable
  but redundant), but the table also enumerates the *originating
  Java exception class*, which is information the spec does not
  carry and that a server-side reader benefits from. The README is
  not the right place ŌĆö it points at the spec for the contract and
  does not list per-error-code details.

## How this compares to what I know

- **In Scala/Typelevel ŌĆö tapir style.** With tapir you would model the
  error as a sealed ADT: `sealed trait DomainError; case object
  RoomNotFound extends DomainError; ŌĆ”` and either derive a circe
  `Codec` (for JSON shape) or define `tapir`'s `oneOf` mapping with
  status codes per branch. The endpoint definition is a value, so the
  type-level enum and the wire format share one source of truth ŌĆö no
  codegen step. The OpenAPI document is then *derived* from the
  tapir endpoints. Here we are taking the inverse path: Java source
  declares the contract via annotations, Springdoc renders the OpenAPI
  document, and the frontend's codegen derives the TS types from the
  document. Two codegen steps instead of one ADT shared at compile
  time. The trade-off is "we run Java/Spring like Spring, not like
  Scala-with-extra-steps" ŌĆö the spec is the lingua franca instead of
  the ADT.

- **circe derivation parallel.** The String-literals-in-annotation
  choice maps cleanly to circe's `Encoder/Decoder` for a sealed ADT
  with `deriveEnumerationEncoder` / `Decoder` ŌĆö both encode to a
  closed set of strings. The decisive difference is that circe gives
  you a *Scala* compile-time guarantee (the ADT cannot have a value
  outside the set), whereas the `@Schema(allowableValues)`
  declaration is *advisory* at the Java compile level ŌĆö the handler
  could still emit `"BANANA"` and the JVM would not complain. The
  drift canary closes that gap at the test level. A Java enum + a
  custom Jackson serializer would be the closest analogue to the
  Scala approach; we chose against it (see Decisions).

- **`openapi-typescript` parallel in Node.** `openapi-typescript` is
  effectively the Node ecosystem's "give me TypeScript types from
  this spec" tool; the closest Scala equivalent is `sttp.openapi`'s
  schema readers. Both are post-hoc readers of an OpenAPI document
  that the server produced through a separate mechanism. The
  contrast with tapir is fundamental: tapir treats the endpoint as
  a value at compile time; `openapi-typescript` treats the spec as a
  serialized contract at build time. Both end at the same place
  (typed client) through different doors.

## Gotchas / things I learned the hard way

- **The existing `@Schema` on the `error` component had a different
  description and slightly different formatting.** The original prose
  said "Stable upper-snake-case error code; intended for programmatic
  matching." The plan's description is more informative ("identifying
  the error class. Intended for programmatic matching by clients"); I
  used the plan's wording.

- **The drift-canary test must verify the enum exists.** A first cut
  could just have called `enumNode.elements()` and compared the list
  ŌĆö but if springdoc silently fails to pick up `@Schema(allowableValues)`
  (e.g. due to a Jackson scanning quirk), `enumNode` would be a
  missing-node, `elements()` would yield empty, and a sorted empty
  list compared against the expected sorted list of nine would fail
  ŌĆö but with a confusing message ("expected 9, got 0") that doesn't
  point at the root cause. The explicit `assertThat(enumNode.isArray()).isTrue()`
  with an `.as(...)` description fails first if the annotation is
  ignored, and only checks the contents if the array exists.

- **`@Schema(allowableValues = {...})` is a String array literal in
  source.** Lining up nine String literals inside an annotation
  argument list (which Spotless will reformat) is not pretty. Putting
  one per line, indented under the opening brace, is the readable
  shape and survives Spotless.
