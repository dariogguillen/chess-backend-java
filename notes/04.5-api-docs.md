# Feature 04.5 — API documentation (springdoc OpenAPI)

**Feature ID:** `api-docs` (from `feature_list.json`)

**Status:** in progress

---

## What we built

Adopted `springdoc-openapi-starter-webmvc-ui` 2.8.6 so the application
serves an auto-generated OpenAPI 3 spec at `/v3/api-docs` and an
interactive Swagger UI at `/swagger-ui.html`. Every current endpoint
(`GET /api/health`, `POST /api/rooms`, `POST /api/rooms/{id}/join`) is
annotated with `@Tag`, `@Operation`, and the full set of `@ApiResponse`
status codes the `GlobalExceptionHandler` can produce; 4xx responses
reference a single shared `ErrorResponse` component schema. A new
`OpenApiIT` (3 tests) asserts the spec is well-formed, references
`ErrorResponse` for every 4xx on the room endpoints, and includes a
canary test that fails the build if any future operation ships without
a non-empty `summary`.

## Java / Spring concepts that appear

- **`@RestControllerAdvice` as the producer, `@ApiResponse` as the
  declaration.** The same exception classes that the advice maps to
  HTTP statuses (e.g. `RoomNotFoundException` → 404) must each appear
  as an `@ApiResponse` on every controller method that can throw them.
  springdoc has no way to introspect the advice (it does not parse the
  bodies of `@ExceptionHandler` methods), so the two are kept in sync
  by convention plus the `apiDocs_includesOperationSummaries` canary
  and the per-endpoint per-status checklist in `CHECKPOINTS.md`. See
  the [springdoc FAQ on documented status codes](https://springdoc.org/#how-can-i-extract-fluent-api-from-the-controller-methods).

- **`@OpenAPIDefinition` vs `@Bean OpenAPI`.** Both produce the
  top-level `info` block (title, description, version). We pick the
  bean approach because it composes with constructor injection — the
  bean takes an `ObjectProvider<BuildProperties>` and reads the
  Maven-set version at runtime. The annotation form `@OpenAPIDefinition(info = @Info(version = "..."))`
  takes only a compile-time constant for `version`, which is
  precisely the thing we want sourced from `BuildProperties`. The
  `ObjectProvider` keeps context startup resilient when
  `META-INF/build-info.properties` is absent (the bean falls back to
  `"unknown"`).

- **`@Schema(implementation = …)` as a schema reference.** Each 4xx
  on a controller method declares its body via
  `@Schema(implementation = ErrorResponse.class)`; springdoc emits a
  `$ref: "#/components/schemas/ErrorResponse"` rather than inlining the
  schema. The class-level `@Schema(name = "ErrorResponse", description = …)`
  on the record is what gives the component its name and high-level
  description; the per-component `@Schema` on `error` then refines a
  specific field. See [swagger-core `@Schema` annotation](https://github.com/swagger-api/swagger-core/wiki/Swagger-2.X---Annotations#schema).

- **Springdoc's reflective generation.** springdoc walks the Spring
  application context at startup, finds every `@RestController` bean,
  reflects on its handler methods, and reads the OpenAPI annotations
  it finds. The DTOs in the parameter / return position are pulled in
  by following the types — Jackson annotations, Jakarta Validation
  constraints (`@NotBlank` becomes `minLength: 1`), and OpenAPI
  `@Schema` annotations all contribute. No build-time codegen, no
  separate `openapi.yaml`. The trade-off is that the spec only exists
  at runtime — there is no compile-time check that an endpoint and
  its declared `@ApiResponse` agree.

## Decisions taken

**Inline annotations on controllers and DTOs, not a separate
`OpenApiCustomizer` config class.**

- Decision: every `@RestController` and its DTOs carry their springdoc
  annotations directly at the read site.
- Alternatives: centralize everything in an `OpenApiCustomizer` /
  `GlobalOpenApiCustomizer` bean that builds the spec
  programmatically; mix the two (annotations for shape, a customizer
  for cross-cutting concerns).
- Why: discoverability. A reader looking at `RoomController.joinRoom`
  sees the contract right there — status codes, body schemas,
  case-insensitive path note. Indirection through a customizer would
  make the contract live in two places. Trade-off: controllers get
  noisier; revisit when more than ~10 endpoints with complex error
  matrices land.

**`ErrorResponse` as a shared component schema.**

- Decision: the record carries a class-level
  `@Schema(name = "ErrorResponse", description = …)` and every 4xx
  `@ApiResponse` across the API references it via
  `@Schema(implementation = ErrorResponse.class)`.
- Alternatives: inline the schema in every `@ApiResponse`
  (duplication, drift); split into per-error-code schemas (e.g.
  `RoomNotFoundResponse`) so the spec carries the discriminator
  values — overkill for a project with one error envelope shape.
- Why: one source of truth on the wire, one source of truth in the
  spec. New 4xx error codes just become new error values, not new
  schemas.

**Selective `@Schema` on DTOs.**

- Decision: add `@Schema` only where the field name and type do not
  already convey the contract — alphabet rules on `roomId`, UUID
  format on `playerId` / `gameId`, enum-like values on `role`,
  nullable semantics on `gameId`, useful examples on `displayName`.
  Trivial fields stay unannotated.
- Alternatives: blanket `@Schema(description = …)` on every record
  component for consistency; the inverse — skip `@Schema` entirely
  and rely on JavaDoc and field names.
- Why: the spec carries the information that a Swagger UI reader
  cannot infer from the name alone. Decorating every field would
  generate noise (a `@Schema(description = "the display name")` on a
  field called `displayName` adds nothing). The
  `apiDocs_includesOperationSummaries` canary enforces
  `@Operation(summary = …)`; the equivalent for `@Schema` is not
  added — it would produce false positives on trivial fields.

**WebSocket / STOMP is out of springdoc's scope.**

- Decision: feature 6 will document the WebSocket surface in the
  README, not in the OpenAPI spec.
- Alternatives: AsyncAPI for STOMP (a parallel spec format), or
  documenting STOMP destinations in the OpenAPI `extensions`.
- Why: springdoc only covers REST controllers; the AsyncAPI tooling
  for Spring is far less mature than springdoc, and a portfolio
  project does not need the second spec format. The README will
  carry the STOMP destinations and message shapes when feature 6
  lands.

**README refactor: drop the per-endpoint contract table.**

- Decision: replace the manual method/path/body table in the README
  with a one-paragraph "API documentation" subsection pointing at
  `/swagger-ui.html` and `/v3/api-docs`, plus one curl example per
  endpoint for paste-and-run convenience.
- Alternatives: keep the table and let it drift; auto-generate the
  table from the spec at docs-build time (no docs build today).
- Why: the drift the table was producing is exactly what this
  feature exists to fix. Curl examples stay because they have value
  outside of Swagger UI; the contract details do not, because the
  spec is now the source of truth.

## How this compares to what I know

- **springdoc vs tapir + `OpenAPIDocsInterpreter`.** Tapir treats an
  endpoint as a value — `Endpoint[I, E, O, R]` — and the OpenAPI
  interpreter is just one of many things you can do with that value
  (you can also derive a server, a client, or a docs site). The
  source of truth is the `Endpoint` definition; the interpreter is a
  pure function from that to a Swagger document. springdoc inverts
  the relationship: the source of truth is the running application's
  controller graph, and springdoc reflects on it at startup to
  produce the spec. The same Swagger document comes out either way,
  but the tapir approach gives you compile-time guarantees that the
  endpoint definition and the server match (they are derived from
  the same value), whereas the springdoc approach has only runtime /
  reflection — an `@ApiResponse(responseCode = "404")` that does not
  correspond to any thrown exception is silently allowed.

- **`@Schema(implementation = X.class)` vs circe's implicit
  derivation.** In circe, one canonical `Encoder[A]` is brought into
  scope (via `Encoder.AsObject.derived` or a `derives` clause) and
  every site that encodes an `A` uses it. The compiler enforces a
  single resolution. springdoc's analogue is reflection-based: every
  site that mentions `@Schema(implementation = ErrorResponse.class)`
  contributes the same `$ref` to `#/components/schemas/ErrorResponse`,
  and springdoc deduplicates by class. The one-definition-many-uses
  pattern is identical; the lookup mechanism differs (annotation
  reflection vs. implicit / given resolution).

- **`@Bean OpenAPI` vs configuring an `http4s` `RoutesBuilder`.** In
  http4s you typically build the routes value and pass it to the
  server builder; if you want a Swagger spec, tapir's
  `SwaggerInterpreter` is one composable layer on top of that value.
  springdoc is closer to "annotate-and-let-the-framework-find-it":
  the `@Bean OpenAPI` only sets the root info, and springdoc handles
  the rest by walking the Spring context. The Spring way trades
  composability for less ceremony at small scale.

## Gotchas / things I learned the hard way

- **springdoc 2.8.x targets Spring Boot 3.5.** *(2026-05-18)* The
  README and milestone tracker on the springdoc repo lists 2.7.x as
  the compatible baseline for Boot 3.4; 2.8.x is the version that
  follows Boot 3.5. Pinning to 2.8.6 (latest stable at feature
  implementation time, March 2025) was straightforward — the
  starter brings in `springdoc-openapi-ui` and the swagger-core
  artifacts transitively. No `<exclusions>` needed; Maven's
  nearest-wins did not bite this time.

- **`nullable = true` on `gameId` is the right way to signal "may be
  null".** *(2026-05-18)* Jackson serializes a `null` field by
  default, but the OpenAPI 3 spec needs `nullable: true` on the
  schema to declare it formally. springdoc carries the `@Schema`
  `nullable` attribute straight into the generated spec. Without it,
  a strict spec validator (or a client codegen tool) treats `null`
  as a contract violation even though Jackson is happy to emit it.

- **`/swagger-ui.html` is a redirect.** *(2026-05-18)* The default
  springdoc config serves `/swagger-ui.html` as a 302 to
  `/swagger-ui/index.html`. The IT asserts both: status 3xx +
  `Location` on the first call, then 200 + `Content-Type: text/html`
  on the final URL. This is springdoc's default behaviour and not
  worth changing — but a naïve test that only hits `/swagger-ui.html`
  and expects 200 will fail.

- **`@Schema(name = …)` on the record class makes the component name
  stable.** *(2026-05-18)* Without it, springdoc falls back to the
  simple class name, which happens to also be `ErrorResponse`, so the
  test passes either way. Declaring it explicitly is the defensive
  move: if the record is renamed or moved, the spec's component name
  stays stable for any client that has been generated against it.

- **The canary test walks every path × method in the spec.**
  *(2026-05-18)* `apiDocs_includesOperationSummaries` does not
  hard-code the three known paths — it discovers them at test time.
  When feature 5 adds `POST /api/games/{id}/move`, the test will
  automatically cover the new endpoint with zero changes.

- **Spotless reformatted `@Schema` blocks.** *(2026-05-18)* Inline
  `@Schema(description = "...", example = "...")` on a record
  component was wrapped across multiple lines by Spotless because
  the line exceeded the 100-column soft limit. The result is wordy
  but consistent; do not fight it.

## To dig deeper

- [springdoc-openapi documentation](https://springdoc.org/) — start
  here for configuration properties (`springdoc.api-docs.path`,
  `springdoc.swagger-ui.enabled`, etc.) and migration notes between
  2.x versions.
- [Swagger / OpenAPI annotations reference](https://github.com/swagger-api/swagger-core/wiki/Swagger-2.X---Annotations)
  — the canonical list of `@Schema`, `@Operation`, `@ApiResponse`,
  `@Tag`, and their attributes.
- [OpenAPI 3.0.3 specification](https://spec.openapis.org/oas/v3.0.3) —
  the document format itself, especially the "Schema Object",
  "Response Object", and "Path Item Object" sections.
- [tapir's OpenAPI interpreter](https://tapir.softwaremill.com/en/latest/docs/openapi.html)
  — the Scala-side comparison point, for when you want the same
  output but with the endpoint-as-value model.
