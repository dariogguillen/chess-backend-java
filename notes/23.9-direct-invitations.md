# Feature 23.9 ‚Äî Direct invitations

**Feature ID:** `direct-invitations` (from `feature_list.json`)

**Status:** in progress

---

## What we built

Authenticated users can now invite an accepted friend to a FRIEND room
they created. The invitee gets the invitation in real time over a new
per-user STOMP destination (`/user/queue/invitations`) and can also list
pending invitations over REST (so an offline invitee sees them on next
load). Accepting performs the room join **server-side** using the room's
stored join token ‚Äî the token never reaches the invitee's client ‚Äî and
the inviter is notified by the existing `ROOM_JOINED` event, no new
signal. Invitations are ephemeral (Redis, tied to the room's 24h TTL, no
migration) and re-validated against the live room at read/accept time.

## Java / Spring concepts that appear

- **Per-user STOMP destinations (`convertAndSendToUser` + user-destination
  prefix).** This is the project's first non-broadcast STOMP path. The
  broker gets `/queue` added (`enableSimpleBroker("/topic", "/queue")`)
  and a user-destination prefix (`registry.setUserDestinationPrefix("/user")`)
  in `WebSocketConfig`. A client subscribes to `/user/queue/invitations`;
  Spring's `UserDestinationMessageHandler` rewrites that to a
  session-private queue. The server pushes with
  `SimpMessagingTemplate.convertAndSendToUser(userId, "/queue/invitations",
  event)`, and the `DefaultUserDestinationResolver` maps `userId` ‚Üí
  the user's active session(s). See
  [Spring's "User Destinations"](https://docs.spring.io/spring-framework/reference/web/websocket/stomp/user-destination.html).
- **The principal must reach `SimpUserRegistry`, not just the message.**
  This was the load-bearing gotcha (see below). `convertAndSendToUser`
  resolves a user's sessions through `SimpUserRegistry`, which is
  populated from `SessionConnectEvent` ‚Äî and that event's principal comes
  from the *WebSocket session*, not the STOMP message. `StompSubProtocolHandler`
  registers a `setUserChangeCallback` on the inbound CONNECT's **mutable**
  accessor; calling `setUser` on *that* accessor propagates the principal
  onto the session. Feature 20's interceptor was setting the user on a
  `StompHeaderAccessor.wrap(...)` *copy*, which fed `@MessageMapping`
  principal resolution but not the registry ‚Äî so I changed the CONNECT
  handler to mutate the existing accessor
  (`MessageHeaderAccessor.getAccessor(message, StompHeaderAccessor.class)`)
  when it is mutable, with the wrap-and-rebuild kept as a defensive
  fallback.
- **Redis hashes as a per-key index (`opsForHash`).** `RedisInvitationStore`
  keeps one hash per invitee at `invitations:user:{inviteeUserId}` whose
  fields are room ids. `findByInvitee` is a single `HGETALL` ‚Äî no
  `SCAN`-by-pattern over the keyspace ‚Äî and the field-keyed layout gives
  "at most one invitation per (invitee, room)" for free. TTL is set on
  the whole hash with `redisTemplate.expire(key, ttl)` after each write,
  mirroring `RedisRoomStore`'s `save`-refreshes-TTL contract.
- **Mechanical error-code derivation.** `InvitationNotFoundException
  extends NotFoundException` (‚Üí 404) and `NotRoomMemberException` (‚Üí 403,
  wired the same narrow way as `InvalidJoinTokenException`) both get their
  `error` code from `GlobalExceptionHandler.codeOf` ‚Äî simple class name
  minus `Exception`, camelCase ‚Üí UPPER_SNAKE_CASE ‚Äî so no new branch was
  needed beyond the 403 handler. The `ErrorResponse.allowableValues` enum
  and the `OpenApiIT` canary moved to the exact 21-code set.
- **`@Transactional(readOnly = true)` applied narrowly.** Only the
  friend-gate check touches JPA, so only `InvitationService.send` carries
  the annotation. The Redis store ops and the STOMP pushes are not
  JPA-transactional, so the accept/decline/cancel/list methods carry no
  transaction.

## Decisions taken

- **Decision:** Accept reuses `RoomService.joinRoom` and emits no new
  event for the accept.
  - **Alternatives considered:** a dedicated `InvitationAcceptedEvent` on
    a topic, or returning the token to the client to let it call
    `POST /api/rooms/{id}/join` itself.
  - **Why this one:** the join already produces `RoomJoinedEvent` on
    `/topic/rooms/{roomId}` ‚Äî the inviter is already subscribed there
    after creating the room ‚Äî so a second event would be a redundant
    signal for the same fact. Performing the join server-side also keeps
    the join token off the wire entirely, which is the security point of
    feature 22.7.

- **Decision:** Invitations are ephemeral in Redis tied to the room,
  with no Flyway migration.
  - **Alternatives considered:** a durable `invitations` table mirroring
    `friendships`.
  - **Why this one:** an invitation is only meaningful while its room is
    joinable, and the room itself is ephemeral (24h Redis TTL). A durable
    table would accumulate dead rows for rooms that TTL'd out, and would
    need its own cleanup. Keying per invitee in a Redis hash, with
    liveness re-validated against the live room at read/accept time and
    stale entries pruned lazily, makes "the room is gone" and "the
    invitation is gone" the same fact with no background job.

- **Decision:** The `side` and `timeControl` in the list response are
  derived from the live room, not stored on the invitation.
  - **Alternatives considered:** snapshot them onto the stored
    `Invitation` at send time.
  - **Why this one:** they are properties of the room, not the
    invitation; deriving them at read time means the stored value stays
    minimal and can never be stale on those axes.

## How this compares to what I know

- **In Scala / Typelevel this would be...** the per-user push is the
  shape of an `fs2.concurrent.Topic` *keyed by user* ‚Äî or a
  `Map[UserId, Queue[F, InvitationEvent]]` where each subscriber owns a
  `Queue[F]`. `convertAndSendToUser(userId, ...)` is "look up that user's
  queue(s) and offer the event"; the `SimpUserRegistry` is the
  `Map[UserId, Set[Session]]` you would maintain by hand. The subtlety I
  hit ‚Äî the principal has to be registered, not just attached to one
  message ‚Äî is exactly the bookkeeping you would do explicitly in the FP
  version: you cannot offer to a subscriber you never registered. The
  ephemeral-Redis-tied-to-room model is what you would express as a
  `Ref`/cache with a TTL rather than a Doobie-backed table; the liveness
  re-validation is a `getOrElse(prune)` on read instead of a foreign-key
  cascade.
- **In Node this would be...** `convertAndSendToUser` is
  `io.to(socketIdForUser).emit("invitation", payload)` once you have a
  `userId ‚Üí socketId` map (the Socket.IO rooms/`io.to(userId)` pattern).
  The Redis hash index is a literal `HSET invitations:user:{id} {roomId}
  {json}` + `HGETALL` you would write against `ioredis`. The
  server-side-join-with-stored-token decision maps to "the server holds
  the secret and the client only sends an intent" ‚Äî the same trust
  boundary the Node prototype got wrong by trusting the client.

## Gotchas / things I learned the hard way

- `convertAndSendToUser` silently no-ops if the user is not in
  `SimpUserRegistry`. The server log said "pushed" (the call returns
  without error) while the client received nothing. The fix was setting
  the principal on the **mutable** inbound CONNECT accessor so the
  `setUserChangeCallback` propagates it to the WebSocket session and thus
  the registry ‚Äî setting it on a wrapped copy is enough for
  `@MessageMapping` but not for user-destination resolution.
- Bot-room creation in an IT needs a mocked `BotEngine`; I sidestepped
  that for the `409 ROOM_FULL` case by filling a normal FRIEND room with
  a real join instead of creating a bot room ‚Äî same not-joinable
  condition, no engine dependency.
