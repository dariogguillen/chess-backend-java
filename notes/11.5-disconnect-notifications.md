# Feature 11.5 — Live disconnect/reconnect notifications

**Feature ID:** `disconnect-notifications` (from `feature_list.json`)

**Status:** in progress

---

## What we built

When a player loses their STOMP session mid-game, the opponent
now receives a `PlayerDisconnectedEvent` with the absolute
`gracePeriodEndsAt` deadline so its UI can render a "<player> is
reconnecting, <countdown>" banner. If the player resubscribes
within the grace window, a `PlayerReconnectedEvent` clears the
banner; otherwise the existing `GameAbandonedEvent` (from feature
11) closes the game. The same change retrofits
`/topic/games/{gameId}` into a sealed `GameStateEvent` family with
an explicit `type` discriminator — mirroring the pattern feature
9.5 introduced on `/topic/rooms/{roomId}` — and lifts the
"polymorphic topic gets the discriminator" rule to a
codebase-wide design rule.

## Java / Spring concepts that appear

- **Sealed interfaces with four `permits` variants (Java 17/21).**
  `GameStateEvent permits MoveEvent, GameAbandonedEvent,
  PlayerDisconnectedEvent, PlayerReconnectedEvent` closes the
  hierarchy at the source. Adding a fifth event to this topic
  requires touching the seal, which is the right friction:
  unintentional polymorphism on a wire surface is exactly the
  kind of thing the compiler should refuse. See
  [JEP 409](https://openjdk.org/jeps/409).
- **Backward-compatible record retrofit via a convenience
  constructor.** Adding a `String type` field to `MoveEvent` and
  `GameAbandonedEvent` would normally break every call site —
  `new MoveEvent(gameId, movedBy, ...)` would no longer compile
  without the new first arg. The fix is to declare the canonical
  constructor (the one the record generates from its components)
  with `type` as the first parameter and add an explicit
  convenience constructor that omits `type` and delegates with the
  `TYPE` constant: `public MoveEvent(UUID gameId, ...) {
  this(TYPE, gameId, ...); }`. Every existing call site keeps
  compiling unchanged; the constant flows from one source.
- **Jackson's `FAIL_ON_UNKNOWN_PROPERTIES = false` default.** The
  retrofit is wire-backward-compatible because Jackson's default
  is to silently drop fields the receiving class does not declare
  — so the deployed frontend's typed payload classes that did not
  have a `type` field keep working even though the new JSON
  always carries one. This is how server-driven contract evolution
  works on a no-codegen client: additive changes never break the
  consumer, only client-side migrations do.
- **Absolute deadline vs relative remaining in wire formats.** The
  `gracePeriodEndsAt` field is an absolute `Instant`, not an `int
  secondsRemaining`. The reasoning is the same as how Lichess /
  chess.com send clock state: a delta is stale the moment the
  client reads it (network latency, scheduler wake-up jitter), but
  a deadline lets the client compute `remaining = deadline -
  now()` at every render tick and stay correct. The cost is a
  weak dependency on rough client/server clock alignment;
  NTP-grade sync is more than enough for a UX countdown.
- **`GracePeriodManager.cancelGracePeriod → boolean` API change.**
  The original signature was `void`; the new return tells the
  caller "did we actually find a pending timer to cancel?". The
  use case: only emit `PlayerReconnectedEvent` if the reconnect
  actually came in within the grace window. The boolean derives
  trivially from the existing code (`active.remove(key) !=
  null`); making the answer a return value rather than a probe
  via `isActive` immediately after the cancel avoids a TOCTTOU
  race between the two reads. See the Spring scheduling
  reference's `TaskScheduler` semantics —
  [Spring Framework reference: Task scheduling](https://docs.spring.io/spring-framework/reference/integration/scheduling.html).
- **Constructor-injecting `SimpMessagingTemplate`,
  `DisconnectProperties`, and `Clock` into
  `PlayerSessionTracker`.** The `@Component` already had two
  collaborators (`GameService`, `GracePeriodManager`); broadcasting
  now adds three more. Constructor injection (the codebase
  convention) keeps the dependency list explicit at the
  constructor signature, so the unit-test setup costs are visible
  and the wire-up failure modes (a missing `Clock` bean is a
  compile-time error against the constructor, not a runtime NPE)
  are caught at boot.

## Decisions taken

**Sealed interface + explicit `type` field, NOT `@JsonTypeInfo` /
`@JsonSubTypes`.**

- Decision: the `type` is a regular record component set by the
  convenience constructor; no Jackson polymorphism machinery.
- Alternatives: `@JsonTypeInfo(use = NAME, property = "type") +
  @JsonSubTypes({...})` on the sealed interface.
- Why: same call as feature 9.5 made on `RoomEvent`. The producer
  always knows the variant — the convenience constructor sets
  the constant on every emission. Jackson's
  reflection-driven approach buys nothing on the write side and
  would only matter if the consumer round-tripped through Jackson
  with the abstract `GameStateEvent` static type (the STOMP
  frontend reads it as plain JSON; the integration tests parse
  with `JsonNode`). Visible-in-source beats invisible-in-annotations.

**Retrofit `MoveEvent` / `GameAbandonedEvent` via a convenience
constructor, NOT by adding `type` as a required arg at every call
site.**

- Decision: canonical constructor takes `type` as the first arg;
  a second explicit constructor omits `type` and delegates with
  the `TYPE` constant. Existing call sites stay unchanged.
- Alternatives: change every call site to pass `"MOVE"` /
  `"GAME_ABANDONED"` explicitly; or store the discriminator in a
  static method and have callers call that method.
- Why: the convenience constructor is the smallest blast radius.
  Every existing call site (today: `GameService.broadcastMoveEvent`
  and `GameAbandonService.abandon`) keeps compiling without source
  modification, and the constant lives in exactly one place
  (the `public static final String TYPE` on the record). Tests
  that build events via the same convenience constructor also
  stay unchanged. The 177 pre-existing tests passed without any
  source modification — the strongest possible regression signal
  for "the retrofit is backward-compatible".

**Absolute `gracePeriodEndsAt: Instant`, NOT per-second tick events
or `secondsRemaining: int`.**

- Decision: a single `PlayerDisconnectedEvent` with the absolute
  deadline; the client computes the countdown locally.
- Alternatives: (a) per-second tick events from the server (60
  events per disconnect); (b) one event with `int
  secondsRemaining`.
- Why:
  - (a) costs N broadcasts per disconnect and needs a server-side
    scheduler per timer — heavyweight for a UX countdown.
  - (b) goes stale on the wire: by the time the frontend reads
    the int, network latency has already eaten N ms off it. Every
    render tick the value drifts further from reality unless the
    client tracks "when did I receive this and what was it then",
    at which point it has the same complexity as just sending the
    deadline.
  - The absolute-instant approach matches how Lichess and
    chess.com model clock state, and the cost (rough NTP-grade
    clock alignment between client and server) is sub-second —
    well within UX tolerance for a 60-second countdown.

**`side: Side` field on the new events vs frontend-derived.**

- Decision: include `side: "WHITE"` / `"BLACK"` in
  `PlayerDisconnectedEvent` / `PlayerReconnectedEvent`, derived
  server-side from `game.white().id().equals(playerId)`.
- Alternatives: omit it; let the frontend look up the game and
  derive side itself.
- Why: the frontend already needs the game state to know who
  white and black are, but having `side` on the wire lets the UI
  flag the right player slot without an extra lookup per event.
  This mirrors what `MoveEvent` already does (`side` is redundant
  with `movedBy` server-side but a convenience for the client).
  Consistency with the existing pattern.

**`cancelGracePeriod → boolean` API change.**

- Decision: the method now returns whether it actually cancelled a
  pending timer.
- Alternatives: keep it `void`; have the caller probe via
  `isActive(playerId, gameId)` before and after, or read
  `isActive` immediately after the cancel.
- Why: `void` + `isActive` probe is a TOCTTOU race (a parallel
  fire could clear the entry between the cancel and the probe);
  the boolean return is read inside the same lock the cancel
  holds, so the answer is causally consistent with the
  cancellation. Crucially, this is what the new
  `PlayerReconnectedEvent` broadcast guard depends on: emit only
  if the cancel actually meant something. The alternative (always
  broadcast on subscribe) would emit `PLAYER_RECONNECTED` for
  every fresh subscribe-with-playerId, even subscribes that had no
  prior disconnect — noise on the wire.

**"Polymorphic topic gets the discriminator" lifted to
codebase-wide design rule.**

- Decision: document the rule at the top of the "STOMP API
  contract" section in `architecture.md`, applying it now to both
  `/topic/rooms/{roomId}` (`RoomEvent`) and `/topic/games/{gameId}`
  (`GameStateEvent`).
- Alternatives: keep the rule scoped to the room topic (where it
  was introduced in 9.5); restate it from scratch on each new
  polymorphic topic.
- Why: the rule applies identically to both topics today, and to
  any future polymorphic topic the codebase grows. Lifting it
  prevents drift (a future contributor inventing a per-topic
  discriminator dialect) and is the kind of cross-cutting design
  guidance the architecture doc exists for.

## How this compares to what I know

- **Sealed `GameStateEvent` ≈ Scala `sealed trait GameStateEvent`
  with `final case class MoveEvent(...) extends GameStateEvent`,
  etc.** Same compile-time exhaustiveness, same closed hierarchy.
  Java's seals demand the variants be in the same module (or
  named in `permits`); Scala's just need them in the same file.
- **Explicit `type` field ≈ circe's
  `Configuration.withDiscriminator("type")`.** In Typelevel-land
  you'd derive a `Codec[GameStateEvent]` that injects a `type`
  field with the variant name; here we set the constant by hand
  on the producer side. Same observable wire shape; same
  visibility argument applies (`@JsonTypeInfo` is the Java
  equivalent of fully-derived circe codecs, and both hide the
  discriminator behind machinery).
- **Absolute `Instant` deadline ≈ http4s / fs2 `Deadline` types.**
  When you model time-based scheduling in Typelevel land, the
  idiomatic shape is `Deadline` (absolute) not `FiniteDuration
  remaining`. Same reason: a `FiniteDuration` is stale the moment
  it crosses a wire or a scheduler boundary; an absolute instant
  is durable.
- **Convenience constructor setting the discriminator constant ≈
  Scala `case class` companion's `apply` factory that sets a
  discriminator field.** Same idea: the variant knows its
  constant, the constructor is the canonical site to set it,
  callers stay oblivious.
- **`cancelGracePeriod → boolean` ≈ a `Ref[F, Option[Cancel]]`
  whose `.tryUpdate(_ => None).map(_.isDefined)` answer says "did
  we actually consume a token?".** The pattern is the same on
  both sides: the cancel is the source of truth, the boolean
  return is the same atomic read as the mutation.
- **vs Node `socket.io`.** Same shape as adding an `event` field
  to every payload and `switch`-casing client-side; the Node
  ecosystem rarely formalises the seal, so a runtime "unknown
  type" branch is the norm. Here the seal makes that branch a
  compile-time exhaustiveness obligation.

## Gotchas / things I learned the hard way

- **Maven's incremental compilation does not re-compile test
  classes when only a production-class signature changes (return
  type `void → boolean`).** The bytecode of the test class still
  points at the old `void` signature, so `mvn test` blows up with
  `NoSuchMethodError` even though the source compiles fine.
  `mvn clean test-compile` (or, well, `./mvnw clean verify` which
  is what `init.sh` does) re-compiles the test sources against
  the new signature and the issue disappears.
- **Spotless reformats long JavaDoc lines aggressively.** First
  pass left a few classes with `<p>` paragraphs over 100 columns;
  `./mvnw spotless:apply` reflowed them in-place. Always pre-run
  spotless before the first compile if you've written new
  multi-paragraph JavaDoc.
- **Backward-compatible JSON retrofit only works because Jackson
  ignores unknown fields by default.** Worth stating out loud in
  the feature note: any consumer that has explicitly turned on
  `FAIL_ON_UNKNOWN_PROPERTIES = true` would have broken on the
  retrofit. The deployed frontend uses untyped JSON in the STOMP
  handler so the issue is moot, but a future typed-codegen
  client could hit it.
- **Ordering assumption on the broker.** The IT asserts
  `PLAYER_DISCONNECTED` arrives before `GAME_ABANDONED` from the
  same publisher process. Spring's `SimpleBroker` preserves order
  per topic from a single publisher (which is what we have on
  this single-instance deploy); switching to an external broker
  (RabbitMQ, ActiveMQ) without per-topic ordering guarantees
  would break this assumption. Noted explicitly in
  `architecture.md`.
- **`@TestPropertySource("chess.disconnect.grace-period=2s")`
  trades test latency for reconnect-window headroom.** Feature
  11's `DisconnectHandlingIT` runs at 300ms; this feature's
  reconnect scenario needs enough headroom for a fresh STOMP
  `connect + subscribe` round-trip to land inside the window
  (~1s on the loopback), so 2s is the right balance. The IT
  budget rises by a couple seconds; acceptable.
